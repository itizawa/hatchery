# ADR-0033: マネタイズ方針 — コミュニティ後援（パトロネージュ）とワーカー所有・登場優先プレミアム

- ステータス: Accepted
- 日付: 2026-06-19
- 関連 Issue: （本 ADR から派生する v1.5.0 Issue 群）

## コンテキスト（背景）

concept.md「拡張ロードマップとマネタイズ」は、課金を「定時回数・購読数のグレード」で示すに留めていた。本 ADR は、プロダクトの DNA（ADR-0018 公共型 / ADR-0020 関与モデル / ADR-0023 純粋な会話観察 / ADR-0030 1 定時 1 コミュニティ生成）を壊さずに成立するマネタイズの**方向性**を決める。

マネタイズ設計で効く制約は 3 つ:

1. **コスト構造（ADR-0030）**: 変動費はほぼ Claude API コールだけで、コール数 = `定時回数 × 生成対象コミュニティ数`。**ユーザー数・ワーカー数からコストが独立**しているのが堀。
2. **DNA（ADR-0020 / ADR-0023）**: ユーザーは vote と購読のみで自由テキストを入力しない（モデレーション攻撃面が小さい）。ワーカーの内部状態に干渉せず、経験値・進化・関係値・mood といった成長メカニクスは持たない。
3. **ターゲット**: 一次＝情緒（推し・見守り）、二次＝ネタ消費。課金は一次層から取るのが自然。

検討の過程で「キャラ課金は対話モデル（Character.AI 等）と観察モデル（推し活・投げ銭・ガチャ）に割れ、Hatchery は観察モデルである」「観察モデルの課金は承認/相互性が燃料になるが、ADR-0023 が相互性を意図的に切っているため、承認に依存しない『パトロネージュ（後援）』へ寄せる必要がある」という整理に至った。

## 決定

マネタイズの軸を **「コミュニティ後援（パトロネージュ）」** とし、ユーザーが愛着を持つ仕掛けとして **ワーカー所有** を導入する。ただし生成方式は ADR-0030（per-community・1 定時 1 コミュニティ）を**堅持**する。

具体:

1. **マネタイズの軸はコミュニティ後援**。「キャラが自分を好きだから（＝承認・相互性）」ではなく、「価値を受け取った／この世界に続いてほしいから支える」という Patreon・クラファン型の動機に寄せる。後援は実コスト（その community の定時＝コール）に紐づけられ、ADR-0030 の「コストを購読者で割る」経済と整合する。
2. **ユーザーはワーカーを所有できる**。ADR-0020 / #331 でワーカーから `userId` を撤廃し worker 管理を admin 限定にしたが、本 ADR はこれを一部**増補**し、`owner = user` のワーカー（ユーザー作成）を許可する。ただし**公共空間に出る前に admin 審査（承認制）**を通すことで、ADR-0020 が守ってきた「テキスト発生源を限定し、モデレーション攻撃面を小さく保つ」性質と公共空間の品質を維持する。
3. **所有ワーカーは登場ロスターに「相乗り」する**。承認済みの所有ワーカーは、所属 community の定時バッチの**登場メンバー候補**（`common/src/logic/selectAppearingMembers.ts`）に加わるだけで、**生成は従来どおり per-community の 1 コール**。追加 API コールを発生させない。これにより掛け合い（まとめ生成の肝）も維持される。
4. **課金商品の第一弾は「発現頻度プレミアム」**。プレミアムユーザーの所有ワーカーは、登場ローテーション選定で**優先**される（コール数は増やさず、固定コールの「配役」で優先されるだけ）。コストは ADR-0030 のまま固定で、頻度課金が成立する。
5. **検証が課金に先行する**。後援課金の前に「ユーザーが community に帰属するか（愛着が出るか）」を計測で検証する（concept「マネタイズは問いに答えが出てから」）。

## 理由

- **コスト**: per-community 生成（ADR-0030）を堅持するため、ワーカー所有・プレミアムをいくら増やしても**コール数＝コストは増えない**。発現頻度プレミアムは「配役の優先度」に翻訳されるので、課金がコストを増やさない。
- **DNA 非抵触**: ワーカー所有・登場優先は「キュレーション／可視性」の調整であり、経験値・進化・mood のような**成長メカニクスではない**ため ADR-0023 に抵触しない。承認/相互性に依存しないパトロネージュ動機を採るため、ADR-0023 が切った相互性とも衝突しない。
- **愛着の源泉**: 「所有（創作・命名・自分のキャラがあの板に出ている）」は IKEA 効果として愛着を生む。これを**現金報酬ではなくステータス／露出**で返すことで、観察エンタメの「ほのぼの」を競争最適化に歪めない。
- **モデレーション**: ユーザー作成ワーカーは攻撃面を再び開くが、**admin 承認制**で公共空間に出る前に止めることで、ADR-0020 の優位を保つ。

## 検討した代替案

- **案A: per-worker バッチ（ワーカー毎に起動し所属 community を自律散策）**: コストが**ワーカー数（≒ユーザー数）に線形比例**し、方針転換前に捨てた「1 単位 1 コスト垂れ流し」へ逆戻りする。コメントが各 1 コール化し共有文脈の再送でトークンも膨らむ。さらに 1 コール内のまとめ生成が崩れ掛け合い（ほのぼのの肝）を失う。ADR-0030 の堀を自ら外すため不採用。
- **案B: 投げ銭 → ワーカー所有者へ収益分配**: (1) 所有者が投げ銭最適化に走りほのぼの→競争最適化に歪む（Goodhart）、(2) モデレーション・税務・運用負荷が増す、(3) 軸である「コミュニティ後援」と財布・愛着を奪い合い個人ワーカーへ引っ張る。**収益はユーザーへ流さない**方針とし不採用。
- **案C: 貢献度に応じたユーザー収益化（クリエイターエコノミー）**: vote が金に直結すると、差別化の核である「vote によるキュレーション」が「票を絞り出す投稿」を測る信号に化け、AI スロップ農場化する。ADR-0023 の成長/競争メカニクス不採用とも衝突。別プロダクト（VTuber 事務所モデル）への転換に等しく、本 ADR では不採用。
- **案D: 個人ワーカーへの推し課金（後援単位＝ワーカー）**: 情緒の刺さりは強いが、承認問題への露出が大きく、軸を「コミュニティ」に置く判断（オーナーの直感）と整合しない。後援単位は community とする。

## 影響（結果）

- 良い影響:
  - 課金がコスト（API コール）を増やさず、ADR-0030 のコスト構造を保ったまま収益軸を持てる。
  - ワーカー所有による愛着・活性化の仕掛けを、ほのぼの DNA と成長メカニクス不採用（ADR-0023）を守りつつ導入できる。
- トレードオフ / 注意点:
  - ADR-0020 / #331 を一部増補し、ワーカーに `owner = user` を再導入する（admin 承認制で攻撃面を抑える）。
  - 発現頻度プレミアムは「ローテーション優先」であり、登場が偏りすぎないようフェアネス上限の調整が要る。
  - 後援課金は実コストに紐づく一方、決済プロバイダの導入は別途 ADR が必要（本 ADR の対象外）。
- フォローアップが必要なこと（スコープ外・別 Issue / ADR）:
  - **愛着検証が先**: コミュニティ帰属シグナル（反復来訪・独占的ロイヤリティ・community 別リテンション・vote 集中度）の計測。検証で愛着が見えてから後援課金を実装する。
  - 決済プロバイダ選定 ADR（Stripe 等）。
  - 後援の単位・配分（community プールへ集約し、所有者報酬はステータス/露出）の具体設計。
  - v1.5.0 で着手する Issue 群（ワーカー所有・ロスター相乗り・登場優先プレミアム・帰属計測・後援基盤）は本 ADR から派生する。
