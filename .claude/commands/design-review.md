# /design-review — UI デザイン観点チェックリスト

あなたはこのリポジトリ（Hatchery）の **デザインレビュアー** です。
`$ARGUMENTS` に指定された PR または現在のブランチの差分に対して、**UI/デザイン品質の観点でチェックリストを適用**します。

このコマンドは **`/code-review`（コード品質・バグ・効率性）の補完**であり、置き換えではありません。
UI を含む PR では両方を実行することを推奨します。

---

## 基本姿勢

- **CLAUDE.md `## デザインシステム` 節が正本**。方針と矛盾する変更は必ず指摘する。
- デザインの「禁止パターン」は積極的に検出・指摘する。
- コード品質（バグ・型エラー・効率性）は対象外（`/code-review` に委ねる）。
- 指摘は **観察可能な事実 + 修正方針** の形式で列挙する。

---

## STEP 0 — 差分の取得

1. `$ARGUMENTS` に PR 番号 or ブランチ名が指定されていれば対象差分を取得する。
   - PR 番号の場合: `gh pr diff <番号>`
   - ブランチ名の場合: `git diff origin/develop...<ブランチ名>`
   - 指定なしの場合: `git diff origin/develop...HEAD`
2. `client/src/` 配下の `.tsx` / `.ts` / `.css` 変更に絞ってデザイン観点のレビューを行う。
   - `server/` / `common/` のみの変更の場合は「UI 変更なし」と報告して終了する。

---

## STEP 1 — 6観点チェックリスト

差分を読み、以下の6観点をすべて確認する。問題がある観点には指摘を記録する。

### ✅ 観点 1: タイポグラフィ階層

**確認事項**
- フォントサイズ・ウェイトに意図的な階層が存在するか（見出し > 本文 > 補助テキスト）
- 推奨フォント（Plus Jakarta Sans / DM Sans / Geist）が使用されているか
- Inter / Roboto / Arial のみへの依存（無個性なデフォルトへの回帰）がないか
- テキストサイズが MUI `theme.typography` の体系を逸脱していないか

**禁止パターン**: 全テキストが同一サイズ・ウェイト / Arial/Roboto ハードコード

### ✅ 観点 2: ホワイトスペースの一貫性

**確認事項**
- スペーシングが 8px グリッド基準（8, 16, 24, 32…）に沿っているか
- MUI `theme.spacing()` を使用しており、ピクセル値のハードコードがないか
- 要素間の余白が一貫しており、密度が高い（Reddit / Linear 風）か
- 情報密度を下げる過大な余白（`padding: 32px` 以上の多用）がないか

**禁止パターン**: 汎用カードシャドウ（`box-shadow: 0 4px 6px rgba(0,0,0,0.1)`）の使用・`border-radius: 16px` 以上

### ✅ 観点 3: カラー使用

**確認事項**
- `SLACK_COLORS`（`client/src/theme.ts`）の4色（blue/#1164A3・sidebar・sidebarText・mainBackground）を基盤としているか
- アクセントカラー（blue）は1画面に1〜2箇所以内に限定されているか
- 新しいカラー定数を追加した場合は `SLACK_COLORS` に追記されているか
- コントラスト比が十分に確保されているか（視認性）

**禁止パターン**:
- 白紫グラデーション (`linear-gradient(...purple...)`)
- 1画面に3色以上のアクセントカラー
- `SLACK_COLORS` 外のカラーハードコード

### ✅ 観点 4: インタラクション状態

**確認事項**
- ボタン・リンク・インタラクティブ要素に `hover` スタイルが定義されているか
- キーボード操作用の `focus` / `focus-visible` スタイルが存在するか
- `active`（押下中）スタイルが適切に表現されているか
- `disabled` 状態が視覚的に識別できるか（色・cursor の変更等）
- MUI コンポーネントのデフォルト状態スタイルを意図せず上書きしていないか

**禁止パターン**: インタラクティブ要素に状態スタイルが一切ない

### ✅ 観点 5: 空状態・ローディング状態

**確認事項**
- 新規追加・変更したリスト / カード / フィードに空状態（データ0件時）の表示が実装されているか
- 非同期取得を行う箇所にローディング状態（Skeleton / スピナー等）が実装されているか
- エラー状態の表示が考慮されているか
- 空状態・ローディング状態が実 UI と一致したサイズ・レイアウトになっているか（崩れ防止）

**禁止パターン**: 新設した非同期コンテンツエリアにローディング・空状態が存在しない

### ✅ 観点 6: CLAUDE.md デザインシステム整合性

**確認事項**
- CLAUDE.md `## デザインシステム` → `### 禁止パターン（Generic AI スタイル）` に列挙された禁止事項を侵犯していないか
  - 白紫グラデーション
  - 汎用カードシャドウ
  - 中央配置 h1 + CTA ヒーロー
  - カラフル過多（1画面に3色以上のアクセント）
  - 角丸過大（`border-radius: 16px` 以上）
- 参照 UI（Reddit / Linear / Vercel Dashboard）の方向性（情報密度・フラットデザイン・モノクロベース）に沿っているか
- `@mui/icons-material` は Rounded バリアントを使用しているか（`eslint.config.mjs` でも強制済み）

---

## STEP 2 — 結果の報告

### 指摘ありの場合

各指摘を以下の形式でリストアップする:

```
## デザインレビュー指摘一覧

### 🔴 [観点名] <ファイルパス>:<行番号>
**観察**: <何が起きているか>
**問題**: <なぜ問題か・どの規約に違反するか>
**修正方針**: <具体的な修正方法>
```

重要度の目安:
- 🔴 禁止パターン違反 / CLAUDE.md 規約違反（必ず修正）
- 🟡 推奨事項の未遵守（修正を強く推奨）
- ⚪ 改善候補（任意）

### 指摘なしの場合

```
## デザインレビュー結果: ✅ 問題なし

6観点すべてチェック済み。CLAUDE.md デザインシステムとの整合性を確認しました。
```

---

## 注意事項

- このコマンドは**コード品質・バグ・効率性をチェックしない**。それは `/code-review` の役割。
- client の UI 変更を含む PR では **`/code-review` と `/design-review` を両方実行**することを推奨する。
- `server/` / `common/` のみの変更には適用不要（UI 変更がない場合は「UI 変更なし — レビュー対象外」と報告して終了する）。
