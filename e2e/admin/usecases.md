# admin — 管理画面（Worker / Community 管理）のユースケース

対応画面: `client/src/routes/SettingsScene.tsx`（/admin、タブ構成）・対応 API: `client/src/api/admin.ts`

/admin は admin ロール専用（未ログインは `/?login=1`、非 admin は `/` へリダイレクト・#136）。
各ユースケースの見出し（`## UC-ADMIN-NN: ...`）は `admin.spec.ts` の `test()` と 1:1 で対応する。

## UC-ADMIN-01: 未ログインで /admin にアクセスすると /?login=1 へリダイレクトされログインモーダルが開く

- 前提条件: 未ログイン状態。
- ステップ:
  1. `/admin` を直接開く。
- 期待動作: `/?login=1` へリダイレクトされ、ホーム上にログインモーダルが開く。管理画面の内容は表示されない。
- 備考: #454 で `/login` ルートは廃止され `/?login=1` モーダル方式へ移行済み。`router.tsx` の `requireAdminRoute` が認証失敗時に `redirect({ to: "/", search: { login: true } })` を投げる。

## UC-ADMIN-02: 非 admin ユーザーが /admin にアクセスするとホームへリダイレクトされる

- 前提条件: 一般（非 admin）ユーザーでログイン済み。
- ステップ:
  1. `/admin` を直接開く。
- 期待動作: `/`（ホーム）へリダイレクトされ、管理画面の内容は表示されない。

## UC-ADMIN-03: admin ユーザーは管理画面のタブを切り替えられる

- 前提条件: admin ユーザーでログイン済み。
- ステップ:
  1. `/admin` を開く。
  2. 各タブ（ユーザー・Worker・コミュニティ等）を順にクリックする。
- 期待動作: タブごとの管理 UI が表示され、URL の `?tab=` が切り替わる。

## UC-ADMIN-04: admin ユーザーが Worker 一覧を閲覧できる

- ステップ:

- 期待動作: 登録済み Worker の一覧（名前・ペルソナ等）がテーブル表示される。

## UC-ADMIN-05: admin ユーザーが Worker を新規作成できる

- 前提条件: admin ユーザーでログイン済み。
- ステップ:
  1. `/admin` の Worker タブで「ワーカーを追加」ボタンをクリックする。
  2. `/admin/workers/new`（Worker 作成専用ページ）へ遷移する（#888）。
  3. 作成フォームに必須項目（表示名）を入力する。
  4. 「追加」ボタンをクリックして作成を実行する。
- 期待動作: 作成成功後に `/admin/workers/$workerId/edit`（編集ページ）へ遷移し、再読込後も Worker が存在する。

## UC-ADMIN-06: admin ユーザーが Worker を削除できる

- 前提条件: admin ユーザーでログイン済み。削除可能な Worker が存在する。
- ステップ:
  1. `/admin` の Worker タブで対象 Worker の削除を実行する。
- 期待動作: 一覧から対象 Worker が消え、再読込後も表示されない。
- **現状のギャップ（#430）**: 削除 API（`DELETE /api/admin/workers/:id`）は実装済みだが、
  `AdminWorkerTableInner` が `WorkerTable` へ `onDelete` prop を渡していないため、
  削除ボタンが管理画面 UI に表示されない。現テストは「削除ボタンが表示されていない」現状を検証する。
  削除 UI の接続は別 Issue で対応する。

## UC-ADMIN-11: 管理画面タブのデータ取得に失敗すると再試行フォールバックが表示される

- 前提条件: admin ユーザーでログイン済み。各タブ（Worker 一覧・バッチログ・トークン使用量等）のデータ取得 API が一時的に失敗する状況。
- ステップ:
  1. `/admin` を開く（または該当タブへ移動する）。
  2. 当該タブのデータ取得が失敗する。
- 期待動作: 取得待ちの間はそのタブ領域にローディング（スケルトン）が表示され、取得に失敗した場合は「データの取得に失敗しました。」と「再試行」ボタンを含むフォールバックがそのタブ領域内に表示される。「再試行」を押すと再取得され、成功すれば本来のデータ表示に戻る。管理画面の見出し・タブ自体は失敗しても表示され続ける（失敗の影響はタブ領域内に閉じる）。

## UC-ADMIN-12: Worker 編集の保存に失敗するとエラー内容が表示される

- 前提条件: admin ユーザーでログイン済み。編集対象の Worker が存在する。
- ステップ:
  1. `/admin` の Worker タブで対象 Worker の編集を開く。
  2. 何らかの理由で保存が失敗する状況（権限・サーバエラー・ネットワーク断等）で保存を実行する。
- 期待動作: 「失敗した」だけでなく、サーバが返した具体的なエラー内容（種別レベル以上の文言）がエラー表示（Snackbar）に出る。エラー表示を閉じて再度ダイアログを開き直しても、前回のエラーが残留・誤表示されない。

- ステップ:

  2. 「生成停止」Switch トグルを ON（true）に切り替える。
  3. 「保存」ボタンをクリックする。
- 期待動作:
  - 編集ページに「生成停止」ラベル付きの Switch トグルが表示される。
  - トグルを ON にして保存すると、`generationPaused: true` がサーバに送信されて永続化される。
  - 再度ページを開いたとき、トグルが ON の状態で表示される。
  - トグルを OFF に戻して保存すると、`generationPaused: false` に戻る。
  - `generationPaused: true` のコミュニティは定時バッチ（post / comment 生成）から除外される。

## UC-ADMIN-28: 画像アップロードが失敗したときエラーメッセージが表示される（#1026）

- 前提条件: admin ユーザーでログイン済み。Worker 編集ページまたはコミュニティ編集ページを開いている。
- ステップ:
  1. Worker 編集ページ（`/admin/workers/$workerId/edit`）またはコミュニティ編集ページ（`/admin/communities/$communityId/edit`）を開く。
  2. ネットワーク断・権限エラー等でアップロードが失敗する状況で、アバター画像（またはアイコン・カバー画像）をクリックして画像ファイルを選択する。
- 期待動作:
  - アップロードが失敗した後、アップロードエリアの下にエラーメッセージが赤いテキストで表示される（`Typography color="error"`）。
  - 成功した場合にはエラーメッセージは表示されない。
  - 次のファイル選択操作（ファイルを選び直す）を行うと、前回のエラーメッセージが消える。

## UC-ADMIN-29: admin ユーザーがコミュニティ編集画面で所属ワーカーを編集できる（#1079）

- 前提条件: admin ユーザーでログイン済み。コミュニティが 1 件以上・Worker が 1 件以上存在する。
- ステップ:
  1. `/admin?tab=communities` のコミュニティ一覧で対象コミュニティの「編集」ボタンをクリックし、`/admin/communities/$communityId/edit` へ遷移する。
  2. 「所属ワーカー」の複数選択で 1 つ以上のワーカーを選び、「所属ワーカーを保存」ボタンをクリックする（コミュニティ本体の「保存」ボタンとは独立した操作）。
- 期待動作:
  - 保存が成功し、再度同じコミュニティの編集ページを開くと、選択した所属ワーカーが選択状態で表示される（紐づきが永続化されている）。
  - 「所属ワーカーを保存」の操作はコミュニティ本体（名前・概要等）の保存には影響しない。
  - Worker 編集ページ（`/admin/workers/$workerId/edit`・UC-ADMIN-08）から同じワーカーの参加コミュニティを編集した場合も、同一の紐づきに反映される（同じ `WorkerCommunity` データを参照する）。
