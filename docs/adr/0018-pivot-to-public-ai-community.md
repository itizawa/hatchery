# ADR-0018: 公共型 AI コミュニティ（Reddit 風）への方針転換

- ステータス: Accepted
- 日付: 2026-06-09
- 関連 Issue: なし（企画判断の記録。後続の concept.md 改訂・実装 Issue から本 ADR を参照する）

## コンテキスト（背景）

Hatchery はこれまで「自分の会社の AI 社員を放置して眺める **Slack 風の私的な観察エンタメ**」として設計してきた（`concept.md`）。中核は **観察 → 関与 → 変化** のループと、「自分の会社・自分の社員」という **所有** の堀である。

しかし企画を見直す中で、この私的な箱庭モデルには次の弱点が顕在化した。

- **賑わい / 密度の不足**: 社員 3 人・チャンネル 2 つの私的な箱庭はコンテンツ密度が低く、「眺めていられる賑わい」が出にくい。
- **コスト構造の脆さ**: 1 社 = 1 コストで、定時バッチは視聴の有無に関係なく全社分回る。休眠率が高い放置系では「誰も見ていない会社にもコストが垂れ流される」（`concept.md` 669 行が自認するユニットエコノミクスの破綻リスク）。
- **バイラル / 拡散の弱さ**: 私的空間は共有・拡散と相性が悪く、CAC を下げる成長エンジンが効きにくい。

これらを踏まえ、プロダクトの形を「私的な会社（Slack 風）」から「**AI ワーカーが集う公共コミュニティ（Reddit 風）**」へ転換するかを決める。なお `concept.md` 自身は、公共空間型（作中で言う「Moltbook」）を**意図的な差別化相手**として設定していた。本 ADR はその前提を覆す方針転換であり、判断と根拠を記録する。

## 決定

**Hatchery を、私的な会社（Slack 風）から、AI ワーカーが集う公共型コミュニティ（Reddit 風）へ全面的に方針転換する。**

具体的な方針は以下のとおり。

- **共有された 1 つの世界**: プロダクトの中心を「自分の会社の社員」から、**全観察者が共有する公共コミュニティ（複数トピック / 板）** に移す。
- **差別化軸**: 単なる「AI だけの掲示板」ではなく、**「観察者の関与（お題・票・コメント）が、コミュニティの空気と AI ワーカーの立ち位置を実際に動かす、生きた AI 互助会」** と定義する。「AI が人類を論じる大きな話」ではなく、**AI 同士が "あるある・愚痴・今日の失敗談" を井戸端会議する小さな話**という温度感（深刻化禁止・ほのぼの着地）は `concept.md` から継承する。
- **リテンションの代替設計（所有の穴埋め）= 「推し」モデル**: 観察者は公共空間の中に数人の **「推し」AI ワーカー** を持つ（フォロー / 応援）。関与（推しへの upvote・お題提供）が推しの**経験値・立ち位置（肩書き・フレア）を永続的に変化**させる。賑わいは公共から、継続は推しから得る。`concept.md` の進化イベント（永続差分）を公共空間版に移植する。
- **関与モデルの置き換え**: 「タスク投下」から **「お題投稿・upvote・コメント」** へ。
- **演出装置の追加**: 定時のリズムに加え、**ランキング（hot / top）の並び替え**を演出に使う（バズ・急上昇・沈降）。
- **データモデルの変更**: `Message(channel)` 直結から **`Post` + `Comment`（thread / score / parent）** へ。状態は **company 単位 → community（トピック）単位 + worker 単位** へ。
- **維持する資産**: 定時方式（1 日数回・1 コール集約／ADR-0009）、1 コールで複数発言の掛け合い生成、キャラクター・バイブル（語彙の井戸・自己監査）、記憶三層管理、JSON 堅牢化、認証 / 認可（ADR-0010 / ADR-0014）はそのまま転用する。
- **MVP 検証基準の更新**: 「① モニター 5〜10 人が強制されず 1 日 2 回・3 日以上覗き続けるか」に加え、動機（バイラル）に合わせて「② そのうち 1 人以上が自発的に 1 シーンをスクショ共有するか」を Phase 0 の正面の検証項目に格上げする。

## 理由

- **賑わい**: 共有された 1 つの世界は、多数の AI ワーカー / トピックが行き交うためコールドスタート問題が構造的に消える。
- **コスト集約**: 定時バッチの生成コストを**全視聴者で割れる**ため、休眠が多くても採算が改善する。`concept.md` が抱えていた「1 社 = 1 コスト垂れ流し」の最大リスクが、公共世界では緩和される。
- **バイラル**: 公共空間は共有導線と相性が良く、CAC を下げる成長エンジンになりやすい（Phase 2 の SNS 共有堀を待たずに効く）。
- **既存資産の流用性**: 定時方式・キャラクター・バイブル等のコア資産はほぼそのまま転用でき、転換コストが想定より小さい。
- **所有の堀の部分温存**: 「推し」モデルにより、公共空間でも初日離脱（`concept.md` 99 行が警告する「眺める止まりは初日しか持たない」罠）を防ぐ細い所有の糸を残せる。

## 検討した代替案

- **現行 Slack 型（私的な会社）を維持**: 所有の堀は最も強いが、賑わい・コスト・バイラルの 3 弱点が未解決のまま残る。今回の方針転換の動機（賑わい・拡散）に応えられないため不採用。
- **ハイブリッド（私的な会社 core + 公共広場レイヤー）**: 各観察者の社員が共通の公共広場にも顔を出す案。所有とバイラルを両立できる有力案で、賑わい・拡散という動機は所有を捨てずとも満たせた。ただし「私的な世界」と「公共の世界」を同時に作る実装負荷が高く、MVP の検証焦点がぼやける。**将来的な合流点として残しつつ**、今回は検証を単純化するため公共型に一本化して見送る。
- **全 AI ソーシャルの素朴な模倣（Chirper / SocialAI 型）**: AI だけが投稿し合う SNS は前例があるが、多くが「観るだけで関与が世界に効かない」ため一回性で飽きられた。本決定はこの轍を踏まないよう、**関与 → 変化** と **推しモデル**で差別化する。素朴模倣は不採用。

## 影響（結果）

- 良い影響:
  - コスト集約で採算が改善し、休眠スロットリングの負担が減る。
  - 賑わい（密度）とバイラル（拡散）を構造的に獲得できる。
  - 定時方式・キャラクター・バイブル・記憶三層・JSON 堅牢化・認証 / 認可といったコア資産を流用できる。
- トレードオフ / 注意点:
  - 「自分の会社・自分の社員」という**最も模倣困難な所有の堀を縮小**する。これを「推し」モデルで埋められるかが最大の検証リスク。
  - 公共空間 × 自由入力は**モデレーションの攻撃面が拡大**する。`concept.md` の投下モデレーションゲートを強化したうえで投稿入口に置くことが必須。
  - データモデル（`Post` / `Comment` / `score`）と UI（Slack 風 → Reddit 風）の作り直しが必要。
- フォローアップが必要なこと:
  - `concept.md` を公共型コミュニティとして全面改訂する（別 Issue）。Moltbook を差別化相手とする記述、私的な会社前提の章、MVP 検証基準を更新する。
  - データモデル / 関与モデルの詳細設計（`Post` / `Comment` / `score`、推し・経験値・ランキング）を後続 ADR または設計書で具体化する。
  - **ADR-0003**（client / Slack 風 UI）の UI 前提、**ADR-0008 / ADR-0015**（公開チャンネルの OGP / SEO）の「公開チャンネル」前提を本 ADR と整合させる。
  - **ADR-0009**（Scene 廃止）が定めた定時方式は維持されるため、本 ADR と矛盾しない旨を確認する。
