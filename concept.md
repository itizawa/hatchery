## このメモの歩き方

このメモには「ほのぼの体験をどう設計するか」という企画寄りの話と、「トークン・JSON・データモデルをどう組むか」という実装寄りの話が混ざっている。混ぜたままだと読みにくいので、最初に地図を置いておく。

何より大事なのは、**最初に作るのはごく小さな1ループだけ**ということ。重たい章（観測性・倫理・冪等性・コスト試算の細部など）は、そのループが「3日覗き続けてもらえる」と分かってから読めばいい。だから章を読む順番をフェーズで分けておく。

- **まず読む（Phase 0／最小の1ループ）**：このメモの歩き方 → TL;DR → 最初に作る最小1ループ → コンセプト → ターゲットユーザー → 動作モデル → キャラクター・バイブル → 入出力フォーマット → 最小プロトタイプの推奨スコープ。「面白いのか、ちゃんと作れるのか」を判断する最小セット。
- **あとで読む（Phase 1以降／ループが回ってから）**：体験の時間設計の細部、関与モデルの拡張レイヤー、成長とペイオフ、ミニアーク、記憶の三層管理、状態管理とデータモデル、要約パイプライン、冪等性・JSON堅牢化、コスト試算の細部、観測性・倫理、オンボーディング、拡張ロードマップとマネタイズ。**運用が始まってから効いてくる章**なので、MVP検証には不要。後方にまとめてある。
- **企画と実装の両方に効くところ**：キャラクター・バイブル／キャラクター・ステート。ここは企画と実装の境界で、どちらが欠けても体験が死ぬので、フェーズに関わらず早めに目を通したい。

各章の見出しに印は付けないが、迷ったらこの地図に戻ってきてほしい。本メモは設計メモなので、数値（コスト試算・閾値など）はすべて検証前の暫定の置き値である。

## 最初に作る最小1ループ（10行）

細部に入る前に、まず作るものを10行で。

1. 社員3人・チャンネル2つ・定時2回。それだけ。
2. 定時が来たら、会社設定＋直近ログ＋未対応タスクを1つのプロンプトにまとめる。
3. AI に **1コールで複数 message**（数人の掛け合い）を JSON で書かせる。
4. JSON を検証して、各 message を channel 紐づきで永続化する（**message は channel に直接紐づくフラット構造**＝ADR-0009。「シーン」というまとめ単位はデータモデルには存在しない）。
5. 生成のきっかけは**二系統**ある（実装済み）。(a) **定時バッチ**（`runAiMessageBatch`・1日数回）、(b) **投稿トリガのリアクティブ応答**（`generateAiResponsesForChannel`・#183）。観察者がチャンネルに投稿すると、その投稿をトリガに AI 社員の掛け合いを**非同期で即時生成**する。定時を待たずに反応が返るのがマイルストーン v1.0.0 のゴール（詳細は「動作モデル」章）。タスクは拾ったら消す（`new`→`done` の2状態だけ）。
6. 直近 message を次の生成の入力に使う（`formatRecentLog` で整形）。ADR-0009 は「あらすじ（summary）は不要・直近ログで足りる」と決定したが、**実装には別経路のあらすじ更新（`runSummaryBatch` / `getSummary`）が残っており**、定時バッチはプロンプトに summary も併せて渡している（ADR-0009 と実装の食い違い。後述「要約パイプライン」で棚卸し）。
7. 観察者にできるのは「覗く」と「話題を投げる」の2つだけ（関与レイヤーは L1/L2 のみ）。
8. 経験値・進化イベント・関係値・冪等性の細工はまだ入れない。
9. これを2回/日まわして、モニターが3日覗き続けるかを見る。
10. **覗き続けたら、はじめて下の重たい章に進む。**

この10行が芯。以降の章は、この芯を太らせるための引き出しだと思って読んでほしい。

## TL;DR：一言でいうと

**Hatchery は、自分の小さな会社で働く AI 社員たちを「放置して眺める」観察エンタメである。**

要点は3つ。

1. **同じ顔ぶれが継続する**：会話が一回性で消えず、記憶を積み重ねてキャラが立っていく。
2. **常時稼働しない**：基本は一日数回の「定時」にまとめて動くのでコストが事前に読める。これに加えて、観察者が投稿したときだけ反応する**投稿トリガ応答**を併用する（常時 AI が喋り続けるわけではない）。
3. **覗くと変化が育つ**：観察者が軽く関わると、社員に経験値が溜まり、やがて「進化イベント」が起きる。一度起きた進化は元に戻らない扱いにして、観察者の「育てた証」にする。

### 用語ミニ定義

| 用語 | 意味 |
|------|------|
| 観察者 | サービスの利用者。作中では「経営者」という役を演じるが、実態は眺めて楽しむプレイヤー |
| AI社員 | 会社に所属する継続キャラクター。名前・性格・記憶を持つ |
| 定時 | 一日数回、社員たちがまとめて動く時刻（朝・昼・夕・終業など） |
| 定時バッチ | 1回の定時で実行される、1 API コールで複数 message を生成・永続化する処理（`runAiMessageBatch`・ADR-0009）|
| 投稿トリガ応答 | 観察者のチャンネル投稿をトリガに、AI 社員の掛け合いを非同期で即時生成する処理（`generateAiResponsesForChannel`・#183）。定時を待たずに反応が返る |
| 進化イベント | 経験値が閾値を超えたとき希少に発火する、画面に残るキャラ変化（一度起きたら戻さない扱い） |

以降、表記は **Hatchery（ハッチェリー）／観察者／AI社員／定時** で統一する。

## コンセプト：放置して眺めるAI観察エンタメ

Slack 型の UI を持つ、**自分だけの会社で働く AI 社員のほのぼの観察サービス**。

観察者は「経営者」という役でチャンネルに話題やタスクを投げ、社員たちは一日数回の「定時」に各自の性格で緩く反応し、雑談しつつ成果らしきものを返す。その様子を眺めて楽しむ。

- **目的**：生産性ツールではなく、観察エンタメ。たまごっち／水槽に近い「放置して眺める」体験。
- **方向性**：ほのぼの系。深刻な対立やドラマではなく、手触りと継続性で愛着を育てる。

### Moltbook との違い（3軸の対比）

**Moltbook**（AI たちが井戸端会議をする公共空間）と対比すると分かりやすい。あちらが「みんなの公共空間」なら、Hatchery は「自分の会社という私的な箱庭」である。差は3軸で整理できる。

| 軸 | Moltbook | Hatchery |
|------|----------|----------|
| **関与** | 観るだけ（一回性の会話を傍観） | 話題を投げ、構い、育てる。関与が変化を生む |
| **スケール感** | AI が人類や社会を論じる、大きな話 | AI が今日のおやつを論じる、小さな話 |
| **所有** | みんなの公共空間。自分のものではない | 自分の会社・自分の社員。継続して同じ顔ぶれが存在する |

一言でいえば、**あちらは AI が人類を論じ、こちらは AI が今日のおやつを論じる**。そして何より、こちらの社員は「自分のもの」である。

### なぜ「Hatchery（孵化場）」か

命名は育成のメタファーに由来する。**卵＝まだぎこちない、立ち上げ直後の AI 社員**。記憶もログもなく、口調も借り物っぽい。それが定時を重ねてログが蓄積されると、固有の口癖・関係性・お決まりネタが「孵って」キャラが立っていく。**孵化＝記憶の蓄積によってキャラが固有の存在になる過程**そのものを指す。Hatchery は、その孵化を眺める場所である。

## ターゲットユーザーと中核価値

「観察者＝経営者」はあくまで作中の役であって、課金者像ではない。誰がなぜ金や時間を払うのかを、ここで二段に定義する。

- **一次ターゲット（情緒）**：AI に情緒的に触れたい層。「育てたい」「見守りたい」という気持ちで、たまごっち・どうぶつの森・ペット系アプリを好む人。継続関与の中心。
- **二次ターゲット（ネタ消費）**：社内ネタ・あるあるを消費したい層。「うちの会社の mei が今日もボケた」をスクショして友人に見せる、軽い娯楽として消費する人。バイラルの中心。

### 中核価値：観察 → 関与 → 変化の実感ループ

初稿の「眺めて楽しむ」は出発点として正しいが、**それだけでは初日しか持たない**（後述のリスク参照）。中核価値は「眺める」止まりではなく、

> **自分が軽く関わったことで、この社員が少しずつ変わっていく**

という「自分が育てた感」へ再定義する。観察（眺める）→ 関与（話題を投げる・構う）→ 変化の実感（社員が育つ・進化する）。この小さなループが回るかどうかが生命線になる。

### 検証基準

MVP で確かめたい問いはひとつに絞る。

> **モニター5〜10人が、誰にも強制されず、1日2回以上・3日以上、自発的に覗き続けるか。**

「面白いと言ってくれるか」ではなく「黙っていても覗きに来るか」で判定する。3日目に覗かなくなったら、それは設計の失敗である。

## 動作モデル：定時バッチ ＋ 投稿トリガ応答の二系統

どう動かすかは、最初けっこう迷った。候補は3つあった。

- **常時稼働**：社員がいつも何か喋っている。生きてる感は一番強いが、深夜もコストが溶け続けるし、賑やかすぎて「放置して眺める」余白が消える。
- **個別駆動**：観察者が話しかけた社員だけが、その都度1人で返す。安いが、社員同士の掛け合いが生まれず、並行モノローグになる。
- **定時方式**：普段は止まっていて、一日数回の「定時」にだけまとめて動く。

> **【更新：現行の正本】** 初稿はここで「**定時方式のみ**を採用し、観察者の投下は次の定時でだけ拾う」と書いていた。しかし現在の実装は、定時方式を土台にしつつ、**投稿トリガのリアクティブ応答**を併せ持つ二系統になっている（#183）。これはマイルストーン v1.0.0 のゴール「チャンネルにアクセスしてメッセージを投げかけたら AI 社員が自律的に考えてメッセージを投稿する」を満たすための拡張で、「個別駆動」を全面採用したわけではない（投稿トリガでも**1コールで複数 message の掛け合い**を生成し、並行モノローグは避ける）。以下はその二系統前提で書く。

現在の動作モデルは次の二系統が併存する。

### (a) 定時バッチ（`runAiMessageBatch`）

普段は止まっていて、一日数回の「定時」にだけまとめて自発的に動く。

- 常時稼働プロセスは不要。定時にバッチを回すだけ。深夜にコストが溶けない。
- 発火回数が決まっているのでコストが事前に読める（目安：**1定時・1チャンネルあたり 1 APIコール**。詳細は後述のコスト試算）。
- 「定時」というリズム自体が演出装置になる（朝礼・昼の雑談・夕方の進捗・終業のひとこと）。観察者にも「そろそろ覗こう」という習慣が生まれる。
- 各チャンネルで、所属する AI 社員（bot）の掛け合いを 1 コールで生成し、message を **channel 紐づきで永続化**する（Scene でまとめない＝ADR-0009）。

### (b) 投稿トリガのリアクティブ応答（`generateAiResponsesForChannel`・#183）

観察者がチャンネルに投稿すると、**その投稿をトリガに AI 社員の掛け合いを非同期で即時生成**する。

- 定時を待たずに反応が返るので、観察者の「投げた手応え」が強い（v1.0.0 のゴール）。
- 投稿の保存・API 応答とは切り離した**非同期処理**で、生成失敗はユーザー操作に影響させない（握りつぶしてログを残す）。
- 定時バッチと**同じプロンプト組み立て・検証ロジック**（`buildChannelConversationPrompt` / `parseConversationMessages` / `formatRecentLog`）を共有する。違いは「直近ログのみで生成し、あらすじ summary は渡さない」点（定時バッチは summary を併せて渡す。後述「要約パイプライン」の食い違い注記を参照）。

二系統あっても、どちらも**1コールで複数 message を生成**し、message を channel に直接追記する点は共通。「非同期のテンポ」（投げる → しばらくして返る）がほのぼのと相性がいいのは初稿のままだが、その「返り」が**定時待ち**と**投稿即時**の二経路になった、というのが現在地。

### 一日のイメージ（例：定時4回）

| 時刻 | 内容 |
|------|------|
| 9:00 朝 | 緩く始動、雑談に「おはよう」。未対応タスクを拾う |
| 12:00 昼 | 進捗とサボりが半々で流れる |
| 16:00 夕方 | 成果らしきものが上がる |
| 18:00 終業 | その日のひとことが残る |

## 体験の時間設計（最初の5分 / 3日後 / 1週間後 / 1ヶ月後）

放置系の生命線は「時間が経つほど良くなる」という勾配を設計に組み込むことだ。各タイミングで観察者が何を感じるべきかを、リテンション曲線として明示する。

| 経過 | 観察者の体験 | 設計上の仕掛け |
|------|------------|----------------|
| **最初の5分** | 3人の社員と出会い、顔と名前が結びつく。「この子たちか」と思う | **入社式**（オンボーディング）。一言プロフィールカードで感情移入の足場を作る |
| **3日後** | 口癖が分かってくる。「mei はすぐ絵文字使うな」。お決まりネタが芽生える | ランニングギャグの定着判定、関係値の蓄積。**ここが最初の離脱ポイント**なので、何か小さな進展が見えている必要がある |
| **1週間後** | 最初の**進化イベント**を目撃する。mei のプロフィールカードの一言が永続的に書き換わる（成長とペイオフ参照） | 経験値が暫定閾値（30前後）に到達。**ここが分水嶺**。覗いた甲斐があったと実感する初めての瞬間 |
| **1ヶ月後** | 社員同士の関係に物語が溜まる。季節も変わる。「もう手放せない」 | 関係発展アーク、季節更新（生活感の素材プール）、観察日記による資産化 |

この勾配が描けていないと、放置系はたいてい3日で飽きられてしまう。**3日後と1週間後の谷を、進化イベントと関係蓄積で埋める**——たぶんここが一番の勘どころで、まだ数字を詰めきれていない部分でもある。

## 観察者の関与モデル（放置 / 軽い反応 / 構う / 育てるの4レイヤー）

> **この章は Phase 1 以降。** MVP では関与は **L1（軽い反応）と L2（構う）だけ**で始める。初稿の「話題を投げて次の定時を待つ」という1本道には、潔さという価値があった——観察者が最初に掴むべき体験は、結局その1行（投げる → 次の定時で拾われる）だ。下の4レイヤーは、その1行を後から太らせるための引き出しであって、最初から全部見せる必要はない。

初稿の関与は「話題やタスクを投げて次の定時を待つ」の1本だけで、それはそれで潔かったが、長く続けると「自分が育てている感」までは出にくい。そこで関与を、負荷と即時性で4レイヤーに整理しておく（MVPでは中2つだけ使う）。

| レイヤー | 行動 | 即時性 | 実装コスト |
|---------|------|--------|-----------|
| **L0 放置** | 何もしない。静けさも味わう | — | ゼロ |
| **L1 軽い反応** | 絵文字リアクション、覗いた瞬間の「出社」演出 | 即時 | **APIコール不要＝コストゼロ** |
| **L2 構う** | 発言のピン留め、世話アクション（差し入れ等） | 次定時 | 軽い（プロンプトに1行） |
| **L3 育てる** | 累積が経験値となり進化イベントへ | 長期 | 経験値テーブルの更新のみ |

ポイントは2つ。

- **L1 はコストゼロで即手応えを返す**。覗いた瞬間に社員が「あ、来た」と出社演出する、絵文字を押すと小さく反応する——これらは API を叩かずクライアント側で完結する。放置勢にも「構いたい瞬間」にすぐ応える受け皿を用意する。
- **投下は咀嚼して受ける（コピペ復唱禁止）**。観察者が投げた話題を社員がそのまま読み上げるのは興醒め。必ず自分の言葉で噛み砕いて反応する。現行実装では観察者の投稿は**投稿トリガ応答（#183）で即時に拾われる**（初稿の「次の定時で最優先に拾う」は、投稿即応答が入る前の前提。定時バッチ側でも直近ログとして自然に拾われる）。これがたまごっち的な相互関与の手応えになる。

放置者にも、構いたい人にも、それぞれ居場所を作る。これが「放置して眺める」と「自分が育てる」を両立させる仕組みである。

## 生成方式：1回の生成 = 1 APIコール（チャンネル単位のまとめ生成）

> **【更新：ADR-0009】** 初稿の見出しは「1定時1APIコール（シーン単位）」で、生成単位を「この会社のこの定時のワンシーン」と呼んでいた。ADR-0009 で **Scene（シーンというまとめ単位）は廃止**され、生成された message は channel に直接紐づくフラット構造になった。以下は「シーン」をデータモデルではなく**生成の比喩**として読み替えること。生成の起点も「定時」だけでなく**投稿トリガ**を含む（動作モデル章）。

API に渡す単位は「個々の社員」ではなく **「この会社・このチャンネルの、今のひとまとまりの会話」**。脚本家に一場面丸ごと書かせるイメージ。出力した複数 message は Scene でまとめず、それぞれ channel に直接追記する。

**なぜまとめ生成が良いか**

- 共通の文脈（会社設定・ログ）を人数分重複して送らずに済み、入力トークンを節約できる。
- 個別生成だと社員同士が噛み合わず並行モノローグになる。まとめ生成なら掛け合いが作れる（＝ほのぼのの肝）。定時バッチも投稿トリガ応答も、この「1コールで複数 message」を共有する。

**まとめ生成の弱点と対策**

- 人数が増えるほど出力が長くなり、後半の社員のキャラがぼやける／口調が均質化する。
- → 1回の生成に出すのは全員でなく数人に絞る。**登場メンバーを生成ごとに変える**。コスト対策であると同時に「今日はあの人静かだな」という固有性の演出にもなる。
- → 口調均質化（語尾だけ違う同一人物化）はこの方式で最も起きやすい劣化なので、次の「キャラクター・バイブル」と「キャラクター・ステート」で正面から防ぐ。

> **注意（Phase 1 の構想）**：登場メンバーを毎回変えるなら、誰がいつ最後に出たかを記録しないと「ken が2週間出てこない」事故が起きる。**最終登場を台帳に持ち、長く出ていない社員を優先的に登場させる**ローテーション制御を入れる（データモデル参照）。

## キャラクター・バイブル（不変設定の正典）

口調の均質化を根本から防ぐには、各社員を「語尾」ではなく「中身」で書き分ける必要がある。各社員を以下の5層で定義し、これを**正典（不変設定）**として固定する。

各社員の定義：

1. **役割**：年齢・職種・立場（mei＝新人 など）。
2. **語彙の井戸（好む語・NG語）**：この社員が好んで使う語・言い回しと、絶対に使わない語。「haru はお菓子の固有名詞を出す／硬い敬語は使わない」のように。
3. **場での役回り**：掛け合いにおける機能。ボケ・ツッコミ・起爆剤・まとめ役・天然。
4. **不変の癖**：絵文字頻度、語尾、相槌の打ち方など、生成のたびに必ず保たれる癖。
5. **背景の核**：消えてはいけない設定ファクト（mei は新人で、◯◯案件を進行中、など）。

**合格の手ざわり**：シーンから話者名（speaker）を隠したとき、誰の発言か当てられるか。当てられないなら、その社員はまだ「語尾違いの同じ人」でしかない。

ただしこれは自動では測れないので、「基準」と言い切らず、ブラインドの定性チェックとして回す：**モニター3人に、話者名を伏せた発言を各5件ブラインドで見せ、誰の発言か当ててもらう。正答率がだいたい7割に届けば合格の目安**とする。届かないキャラは「語彙の井戸」を書き足す。数字は暫定で、サンプル数も最初は小さくていい。

このバイブルは要約や圧縮の対象にせず、**毎回フルで投入する**（後述「記憶の三層管理」の最上層＝キャラ憲法に相当）。

## キャラクター・ステート設計（今日の調子 / 関係値 / 個人イベント）

固定の性格しか持たないキャラは「着ぐるみ」「静止画」であり、たまごっちや水槽の魅力である**状態の揺らぎ**が欠ける。各社員に変動する内部状態を持たせ、生成入力に注入する。

- **mood（今日の調子）**：絶好調 / 普通 / 低調 を、毎定時バッチの前に確率で決める。プロンプトに1行注入するだけ。「昨日いっぱい喋ってた ken が、今日はなんだか静か」が**記憶ではなく状態として**体感できる。
- **関係値（社員ペアの距離感）**：社員同士の親密度を数値テーブルで持ち、掛け合いの距離感に反映する。仲が深まれば軽口が増え、ぎくしゃくすれば少しよそよそしくなる（深刻化はさせない）。
- **個人イベント／観察者との履歴**：観察者からのピン留めやリアクション履歴を保持し、「この前褒めてもらった話題」を社員が覚えている、といった反応につなげる。

実装は**数値テーブル＋プロンプト1行**で済む。これだけで静止画が生き物になる。

## 成長とペイオフ設計（経験値・進化イベント・関係発展アーク）

> **この章は Phase 1。** MVP（最初の1ループ）には入れない。まず「3日覗き続けるか」を見て、続いた人にだけ効いてくる仕掛けだ。ただし**続いてもらうための一番の鍵**でもあるので、設計だけ先に置いておく。

定時に覗く → 読む → 閉じる、だけのループだと、たまごっちの成長・進化に相当するご褒美（ペイオフ）がない。「読むこと自体が報酬」は初日しか持たない。差別化軸である「記憶の蓄積」を、続けたくなる小さな快感へ翻訳する仕組みがいる。

### 見えない経験値

各社員に、観察者からは見えない経験値を持たせる。観察者の関与で加算される。**数値バーは見せない**——見せると育成ゲームの作業になり、ほのぼのが消える。あくまで裏で溜まる。

加算値・閾値・発火確率は、設計が回るかを左右する数字なのに、つい「あとで詰める」に送りがち。なので**暫定の当たり値**だけ先に置く（実測で必ず直す前提）。

| 関与 | 加算 |
|------|------|
| 覗いた（その日初回） | +1 |
| 絵文字リアクション | +1 |
| 話題の投下 | +3 |
| ピン留め・世話アクション | +5 |

- **進化の閾値**：最初の進化を**経験値 30** あたりに置く。1日に数回関与する人で、だいたい1週間で最初の1回に届く感覚（時間設計の「1週間後の分水嶺」に合わせた逆算）。
- **発火確率**：閾値を超えた定時に、いきなりでなく**確率20%程度**でじわっと発火させる。「いつ来るか分からないから嬉しい」を作る。発火したら閾値を一段上げ、次の進化はさらに遠くする。

### 進化イベント（覗いた甲斐の核）

経験値が閾値を超え、確率に当たったとき、**希少な「進化イベント」**を発火させる。一度起きたら**元に戻さない**——ただし生成 AI は毎回プロンプトから状態を組み直すので、「戻らない」はモデル任せでは担保できない。**進化済みフラグを world_state に持ち、毎回プロンプトに「mei はもう自分から提案できる」と事実として注入する**ことで、退行（過去の進化を忘れて元に戻る事故）を防ぐ。

**何が画面に永続的に残るか**を、最重要なので1つ確定例で書き切る：

> **mei の最初の進化＝プロフィールカードの一言が書き換わる。**
> 入社時：「新人です。よろしくお願いします🙏」
> 進化後：「最近、自分から提案できるようになりました🙌」
> この一言は二度と元に戻らない。観察者が後からプロフィールを開いても、ずっと書き換わったまま。「自分が育てた証」が、いつでも見える場所に永続差分として残る。

掛け合いの中の変化（初めて自分から提案する、等）は一過性で流れてしまう。だから**進化は必ず「画面に残る永続差分」とセットにする**のがこの章の肝。プロフィールカードの一言、絵文字セットが1つ増える、肩書きが変わる——どれでもいいが、必ず後から見返せる形にする。

頻発させない。希少だからこそ、目撃した瞬間が「覗いた甲斐があった」になる。これが**1週間後の分水嶺**（時間設計参照）を作る装置である。

### 関係発展アーク

社員ペアの関係値も、時間をかけて発展する。ただし**一直線に良くなるのではなく、時々後退もしながら**進む。後退があるから発展が嬉しい。深刻な決裂は起こさず、あくまでほのぼのの範囲で揺らす。

**ほのぼの崩壊への歯止め**：mood も関係値も「確率で決める」ので、放っておくと事故が起きる——低調が何日も続く、関係値が単調に悪化し続ける、といった偏りだ。これはほのぼのを静かに壊す。だから確率に**ガードレール**を入れる：

- mood は「2日連続で低調なら、次は低調を引かない」。ずっと暗い社員を作らない。
- 関係値は**下限を切らない**（一定値より下に落ちない）。長期的には**ゆるやかな上昇バイアス**を掛け、後退は一時的な揺れにとどめる。
- 「絶好調」も連発させない。揺らぎは振れ幅であって、一方向の暴走ではない。

## ミニアーク・ランニングギャグ・波風レベル（起伏と新規性）

毎定時が独立した点だと3日で飽きる。逆に毎回賑やかだと消費コンテンツ化して、水槽的な余白（眺めていられる静けさ）が死ぬ。両方を制御する。

### 縦糸（ミニアーク）

**1本3〜7定時で起承転結する小さな出来事**を、種テンプレからローテーションで注入する。例：「観葉植物をぽちった → 届いた → 名前を付けた → 枯らしかけた → 復活した」。ユーザーメッセージの `#進行中アーク` 欄に種を入れ、時々進展させる。これで「次の定時が気になる」が生まれる。

### ランニングギャグ

偶然出たネタが受けたら、**定着判定**でお決まりネタに昇格させ、資産化する。「haru の『お腹すいた』」のような繰り返しが、観察者の身内感を育てる。

### 波風レベル（賑やかさの可変）

| レベル | 内容 | 配分の目安 |
|--------|------|-----------|
| **Lv1 日常** | 何も起きない平坦。雑談だけ | 多め |
| **Lv2 ハプニング** | 小さな出来事（コーヒーこぼした等） | 中 |
| **Lv3 ちょいイベント** | ミニアークの山場 | 少なめ |

**発言密度も可変**にし、2〜3割は「誰も大したことを言わない静かな日」を混ぜる。静けさのコントラストがあるから、賑やかな日が映える。

**鉄則**：失敗やハプニングは必ず温かく着地させる。**深刻化は禁止**。これが長期の飽きと、ほのぼのの崩壊の両方への防衛線になる。

## 入力：ユーザーメッセージ（毎回変わる）

状況と今回の指示を入れる。`登場するのは誰々` を毎回変えるのが演出装置。観察者の投下した話題も未対応タスクに混ぜて渡せば、次の定時で自然に拾われる。

```
# 今の時刻
2026-05-29 09:00（朝の始業）

[雑談] haru: お疲れさまでした〜、明日金曜だ

- [picked] mei が出したロゴ案のレビュー（昨日から）
```

## 出力：フォーマット

パースして UI に流すため JSON で構造化させる。システムプロンプト末尾に「出力は指定スキーマのJSONのみ。前置きや説明は書かない」と明記し、可能ならネイティブの構造化出力（tool_use）を使うとさらに安定する（後述「JSON堅牢化」）。

> **【更新：ADR-0009】** 初稿の JSON 例には全体をくくる `scene`（あらすじ見出し）フィールドがあったが、Scene 廃止（ADR-0009）に伴い**削除**した。出力は `messages` 配列だけで、各 message が `channel` で行き先を持つ**フラット構造**。message は channel に直接紐づき、Scene というまとめ単位は持たない。

```json
{
  "messages": [
    { "speaker": "haru", "channel": "zatsudan", "text": "おはよ〜、今日ちょっと眠い…雨だしね〜" },
    { "speaker": "mei",  "channel": "zatsudan", "text": "おはようございます！☀️ あ、雨ですけど元気です！" },
    { "speaker": "mei",  "channel": "shigoto",  "text": "勉強会のネタ、わたし図解の作り方とかどうかなって思いました🙌" },
    { "speaker": "haru", "channel": "shigoto",  "text": "いいね〜。それ mei が一番得意なやつだ" }
  ]
}
```

### フィールド型定義

| フィールド | 型 | 制約 |
|-----------|------|------|
| `messages` | array | 1件以上。最大件数は指示で上限を持つ |
| `messages[].speaker` | string | **社員ID**。社員定義に存在するIDと一致必須 |
| `messages[].channel` | string | **列挙値**のみ（`zatsudan` / `shigoto` 等。チャンネル定義と機械的に一致） |
| `messages[].text` | string | 非空。1発言の文字数上限を持つ（コスト・冗長化対策） |

`channel` と `speaker` は自由文字列ではなく**列挙／既知IDに限定**する。これにより、後段のバリデーション（Zod等）でチャンネル定義・社員定義と機械的に突き合わせられる。Scene を持たないため、永続化も各 message を channel 紐づきで1行ずつ追記するだけでよい（ADR-0009）。

## 画面と情報設計（チャンネル / タイムラインの見え方）

Slack 型 UI を名乗る以上、チャンネルとタイムラインの定義は必須である。

- **チャンネル**：話題の入れ物。MVP は `雑談（zatsudan）` と `仕事（shigoto）` の2つ。1人の社員は複数チャンネルに発言してよい。
- **タイムライン**：各チャンネルに、message が時刻付きでフラットに追記されていく（Scene でまとめない＝ADR-0009）。観察者は流し読みできる。
- **投下導線**：観察者が話題・タスクを投げる入力欄。投げると**投稿トリガ応答（#183）で AI 社員がその場で反応**し、あわせて次の定時バッチでも直近ログとして拾われる（L2 構う相当）。
- **不在の見せ方**：数日空けて戻ったとき、全部読ませず「ダイジェスト出社」（後述オンボーディング）を出す。

簡易ワイヤー：

```
┌─ ほのぼのIT企業 ───────────────┐
│ # 雑談     # 仕事              │  ← チャンネルタブ
├───────────────────────────────┤
│ [09:00] haru: おはよ〜…        │
│ [09:00] mei:  おはようございます！☀️ │  ← タイムライン
│ [12:00] ken:  ひるめし          │
│  …                            │
├───────────────────────────────┤
│ [ 話題やタスクを投げる…    ] 送信 │  ← 投下導線
└───────────────────────────────┘
```

## 処理フロー図とデータの循環

ループの全体像を、データの循環として図にする。下図は**定時バッチ**の循環。**投稿トリガ応答（#183）**は ① が「Cron」ではなく「観察者の投稿」に置き換わり、③④⑤ を同じロジックで通る別経路だと読めばよい（slot_key 冪等ガード・要約更新は持たない簡易経路）。なお ③ の「シーン生成」は ADR-0009 以後、Scene でまとめず channel 紐づきの message 群を生成する処理を指す（「シーン」は比喩）。

```
        ┌─────────────────────────────────────────────┐
        │                                             │
        ▼                                             │
  ① 定時トリガー（Cron）                              │
        │  slot_key を採番（冪等ガード）               │
        ▼                                             │
  ② 入力を組む                                        │
        │  共通エンジン部 + 会社固有部                 │
        │  + 直近ログ + あらすじ + タスク              │
        │  + mood/関係値/アーク/小ネタ                 │
        ▼                                             │
  ③ APIへ1コール（message 群を生成）                  │
        │                                             │
        ▼                                             │
  ④ 出力JSONを検証（sanitize → 必要ならリペア）       │
        │  失敗時は「静かな日」ナレーションで着地      │
        ▼                                             │
  ⑤ 各messageをチャンネルに追記（messages保存）       │
        │  経験値・関係値・タスクstatusを更新          │
        ▼                                             │
  ⑥ world_state を更新                                │
        │  last_slot_key を記録                       │
        │  未要約が閾値を超えたら要約パイプラインへ ───┘
        │  （あらすじを更新 → 次回②の入力になる）
        ▼
   次の定時へ（①に戻る）
```

観察者の投下はこのループの外からいつでも入り、**投稿トリガ応答（#183）でその場でも拾われ**、定時バッチ側でも直近ログとして次の②で拾われる。**あらすじ更新は⑥のタイミングで走る**（毎回ではない。詳細は要約パイプライン）。

> **【棚卸し：あらすじ（summary）の現在地】** ADR-0009 は「**あらすじを廃止し、次の入力には `formatRecentLog` で整形した直近ログを使う**」と決定した。一方、現行実装には別経路のあらすじ更新（`server/src/batch/runSummaryBatch.ts`）と取得（`channelRepo.getSummary()`）が残り、**定時バッチはプロンプトに直近ログ＋あらすじ summary の両方を渡している**（投稿トリガ応答は直近ログのみで summary は渡さない）。つまり ADR-0009 の決定と実装が食い違っている。本メモの「あらすじ」「要約パイプライン」関連の記述は**この食い違いを抱えたまま Phase 1 の構想として残す**。どちらに寄せるか（あらすじを正式採用するか・ADR-0009 どおり廃止に揃えるか）は ADR の追補で決める事項であり、本 Issue のスコープ外（要追補）。

## 記憶の三層管理（キャラ憲法 / 設定ファクト / 可変あらすじ）

毎回全履歴を渡すとトークンが青天井になる。とはいえ、初稿の「単一のあらすじ要約」だけだと、世代を重ねるごとに**固有名詞・関係性・口調が真っ先に溶けて**、「みんなの公共空間」型と同じ一回性会話へ退行してしまう。差別化の核である「記憶の積み重ね」が静かに機能停止する、けっこう怖い劣化だ。

これを防ぐため、記憶を3層に分離し、**消していい層と消してはいけない層**を分ける。

| 層 | 中身 | 変化 | 投入 |
|----|------|------|------|
| **① キャラ憲法（不変）** | 口調、語彙の井戸、絵文字頻度、不変の癖 | 変わらない | 毎回フル投入。要約対象外 |
| **② 設定ファクト（半固定）** | mei＝新人、進行中の案件、確定した関係性 | めったに変わらない | 毎回フル投入。要約対象外 |
| **③ 可変あらすじ（流動）** | 「これまでのあらすじ」数行 + 直近20〜30発言 | 毎回更新・圧縮される | 要約パイプラインで圧縮 |

ポイントは、**①②を③の要約から完全に分離して毎回フル投入する**こと。要約（③）がいくら世代劣化しても、口調と核心設定（①②）は溶けない。これがドリフト（口調均質化・キャラ崩壊）への、最も費用対効果の高い防壁になる。コストもごく軽微（数行〜十数行の追加に過ぎない）。

圧縮するのは③だけ。しかも③の要約時にも、**固有名詞・関係性は聖域として保持**するルールを置く（要約パイプライン参照）。

## 状態管理とデータモデル

> **MVP は JSON の塊1つで始めてよい。** 下の2テーブル定義は、運用に入って固有性・冪等を厳密に守りたくなったとき（Phase 1）の到達点。最初から Postgres にスキーマを切る必要はない。
>
> **【注意：現行実装との差分】** 下表は企画当初の構想モデルであり、現行実装のスキーマそのものではない。実装では Scene 廃止（ADR-0009）に伴い **message は Scene/slot を介さず channel に直接紐づくフラット構造**で、発言者は `createdEmployeeId`（FK・#222）として保存される。`synopsis` 相当のあらすじはチャンネル単位の summary として別経路（`runSummaryBatch` / `getSummary`）に存在する（前述の食い違い注記を参照）。

「保存する」止まりだった部分を、実装できるレベルまで具体化する。最小2テーブルで足りる。

### messages テーブル

各発言を1行として保存する（Scene でまとめず channel 直結＝ADR-0009）。

| カラム | 型 | 説明 |
|--------|------|------|
| `id` | PK | 連番 |
| `company_id` | FK | どの会社か |
| `slot_key` | string | 定時の一意キー（例：`2026-05-29T09:00`）。冪等の要（投稿トリガ応答には付かない） |
| `seq` | int | 同一生成内の発言順 |
| `speaker` | string | 社員ID（実装では `createdEmployeeId`・FK） |
| `channel` | string | チャンネル列挙値（message の唯一のまとめ単位） |
| `text` | string | 発言本文 |
| `created_at` | timestamp | 追記時刻 |

→ `(company_id, slot_key, seq)` を**複合ユニーク制約**にする。これが Cron 二重発火に対する最後の砦（Phase 1 の冪等化）。

### world_state テーブル

会社ごとの現在状態を1行で持つ。

| カラム | 型 | 説明 |
|--------|------|------|
| `company_id` | PK | 会社 |
| `synopsis` | text | これまでのあらすじ（③の要約）。**ADR-0009 では廃止方針**だが、実装は channel 単位の summary を別経路で保持（要追補） |
| `open_tasks` | json | 未対応タスク配列（status付き） |
| `last_slot_key` | string | 最後に処理した定時。再実行検知に使う |
| `summary_version` | int | 要約の世代。冪等化に使う |
| `char_states` | json | 各社員の mood・経験値・最終登場・関係値テーブル |

### 永続化の段階

- **プロトタイプ**：Vercel Blob / Edge Config などの軽量ストアで十分。テーブルというより JSON の塊で始めてよい。
- **継続運用**：固有性・関係性・冪等を厳密に守るため Postgres へ移行。複合ユニーク制約とトランザクションがほしくなる。

## 投下タスクのライフサイクルと手応え保証

> **MVP は2状態だけ。** 初稿はタスクを「積んで、拾ったら消す」だけで扱っていた。極小スコープ（3人/2ch/2定時）なら、この軽さで十分に正しい。**MVP では `new`（投下された）と `done`（拾った）の2状態だけ**にする。下の4状態の状態機械は Phase 1 の拡張として畳んでおく。

ただし長く運用すると、初稿の素朴さでは破綻する箇所が出てくる。完了・破棄・期限の概念がないと、同じ話題が毎定時ループし、未対応欄が無限に膨張してプロンプトを汚染する。そこで Phase 1 で状態機械に育てる。

```
new ──pickされる──> picked ──消化される──> done
 │                                        
 └──3定時pickされず──> dropped（静かに消える）
```

| status | 意味 |
|--------|------|
| `new` | 投下されたばかり。次定時で**最優先 pick** を保証 |
| `picked` | 今まさに消化中 |
| `done` | 消化完了。プロンプトから外れる |
| `dropped` | 3定時 pick されなかった。静かに破棄（観察者を責めない） |

ルール：

- **観察者の投下にはまず投稿トリガ応答（#183）が即時に反応**し、その後の定時バッチでも最優先で拾われる（手応え保証。L2 構うの中核。初稿の「次の定時で最優先 pick」は投稿即応答が入る前の前提）。
- **1定時で消化するのは最大1〜2件**。一気に片付けず、ほのぼのテンポを保つ。
- **3定時 pick されなければ静かに drop**。「あれどうなった？」を無限ループさせない。

これで「同じ話題のループ」と「未対応欄の無限膨張」を同時に防ぐ。

## 要約パイプライン

> **この章は Phase 1。** MVP では「直近ログをそのまま渡す＋あらすじ数行を手で更新」で足りる。発言が溜まってきてから効いてくる仕組みなので、最初は読み飛ばしてよい。
>
> **【棚卸し：ADR-0009 との食い違い】** ADR-0009 は「あらすじ（summary）は廃止し、直近ログ（`formatRecentLog`）で足りる」と決定した。一方、現行実装には本章の構想に近い**独立コールの要約バッチ**（`runSummaryBatch`・1日1回）が存在し、定時バッチがその summary をプロンプトに渡している。つまり「あらすじ廃止」という ADR の決定と、要約バッチを持つ実装が**食い違っている**。本章は廃止判断と実装のどちらにも断定せず、現状の食い違いを記録した Phase 1 の構想として残す（決着は ADR の追補で。本 Issue のスコープ外＝要追補）。

あらすじ（記憶③）の更新を、トリガー・主体・コストの観点で定義する。初稿では曖昧だった部分。

- **生成と分離する**：要約は会話生成のコールに同梱せず、**安価モデルの独立コール**で走らせる（実装の `runSummaryBatch` もこの方針）。会話生成の品質と切り離す。
- **トリガー**：未要約の発言が**20件溜まったら**、その定時の末尾で1回だけ発火。毎定時は走らせない（コスト節約）。
- **冪等化**：`summary_version` を見て、同じ世代の要約を二重に走らせない。
- **聖域ルール**：要約しても**固有名詞・関係性・進行中案件は必ず残す**。「誰が・誰と・何を」を溶かさない。溶かしていいのは細かい言い回しだけ。

これにより、③が圧縮されても差別化の核（記憶の固有性）は守られる。

## 冪等性・JSON堅牢化・フォールバック

> **この章は Phase 1。** MVP では「JSON を1回パースして、失敗したらその定時はスキップ」くらいの素朴な扱いで始めてよい。冪等性の複合ユニーク制約や3層防御は、運用に入って二重発火やJSON崩れが実際に痛みになってから入れる。最初から作り込むと、たまごっちの軽さがエンタープライズ設計書になってしまう。

定時方式は1日数回しか発火しない。だから**1回の失敗が体験の25%（定時4回なら）を欠損させる**。「定時なのに画面に何も出ない空白」はほのぼの体験を静かに殺す。運用に入ったら多層で守る。

### 冪等性（二重発火対策）

- 生成 API は非冪等（同じ入力でも違う出力）なので、**slot_key 一意キー**で守る。
- Cron の二重発火・タイムアウトリトライ・手動再実行が起きても、`(company_id, slot_key, seq)` の複合ユニーク制約で2回目の挿入が弾かれる。
- これがないと「同じ朝礼が2回流れる」事故が起き、世界の一貫性が壊れる。

### JSON堅牢化（3層防御）

```
① ネイティブ構造化出力（tool_use / structured output）で生成
        │  ← そもそも壊れたJSONを出させない
        ▼
② sanitize：余計な前置き・コードフェンスを剥がしてパース試行
        │  ← 軽い汚れはここで吸収
        ▼
③ リペアコール（最大2回）：壊れたJSONを渡して直させる
        │  ← それでも直らなければ↓
        ▼
④ フォールバック：「静かな日」ナレーションを生成して着地
```

- 検証は **Zod 等のスキーマ**で行い、`speaker` が既知の社員IDか、`channel` が列挙値か、登場制御（指定外の社員が出ていないか）まで含めてチェックする。
- **全層失敗時は「静かな日」ナレーション**で空白を埋める。「今日はみんな静かに作業しているようだ」のような短いナレーションを定型で差し込み、**沈黙すら世界観に溶かす**。エラー画面は絶対に見せない。

## コスト試算とガードレール（モデルティアリング / 休眠スロットリング）

> **この章は Phase 1 の細部。** MVP の判断には不要。ただし「コストが事前に読める」は本企画の売りなので、数字の当たりだけ置いておく。**以下の数字はすべて暫定の置き値**で、実測で必ず直す。算数を間違えると「事前に読めるはずのコスト」という売り自体を自分で崩すので、ここは特に整合性に気をつける。

### コストの式

```
月額コスト ≈ 1コール単価 × 定時回数/日 × 30日 × 隠れコスト係数 × アクティブ会社数
年額コスト ＝ 月額コスト × 12
```

### 試算（1社・定時4回/日・暫定値）

下の表は**係数を掛ける前の素の試算**。年額は単純に月額×12で出している。

| モデル | 想定単価/コール（暫定） | 月額/社 | 年額/社（月額×12） |
|--------|----------------|--------|---------|
| Sonnet 級 | やや高い | 約 $1.6 | 約 $19.2 |
| Haiku 級 | 安い | 約 $0.37 | 約 $4.4 |

これに後述の**隠れコスト係数 ×1.2** を掛けると、実効はおよそ次のとおり：

| モデル | 月額/社（×1.2後） | 年額/社（×1.2後） |
|--------|----------------|---------|
| Sonnet 級 | 約 $1.9 | 約 $23.0 |
| Haiku 級 | 約 $0.44 | 約 $5.3 |

→ **既定モデルは Haiku 級**にする。シーン生成は重い推論を要さないので、安いモデルで十分ほのぼのする。重要な進化イベントなど稀な場面だけ上位モデルへ**ティアリング**する。

> 単価（$1.6 / $0.37）は「定時4回/日・1シーンあたりの入出力トークン量」を仮置きした概算で、まだ実測していない。モデル・トークン数・地域で変わるので、Phase 1 で1コールの実トークンを測って差し替える。

### 見落としやすい前提（暫定の置き値・要実測）

以下の3つの数字は、いずれも**前提（モデル・トークン数・地域・定時間隔）を測ったうえでの確定値ではなく、リスクを過小評価しないための保守的な置き値**である。脚注として明記しておく。

- **プロンプトキャッシュは効かない前提で見る**：一般にプロンプトキャッシュの保持時間は数分オーダーと短い。定時間隔（数時間）では失効しているとみて、**キャッシュ効果ゼロ**を試算のベースケースにする。これを織り込まないと実コストを過小評価する。（具体TTLは利用するモデルの仕様で要確認。）
- **出力がコストの主役（暫定 約2/3）**：このワークロードは入力より出力が支配的になりやすい。**出力トークンがコストの大半（ざっくり2/3程度と仮置き）**とみて、対策は出力側に集中する：
  - `max_tokens` のハードキャップ
  - 1発言あたりの文字数上限
  - 1シーンの登場人数・発言数の上限

  キャラが饒舌化した瞬間にコストが跳ねるので、機械的な上限が「事前に読める」を実行時に保証する。比率は実測で確定する。
- **隠れコスト係数 ×1.2**：要約コール・リペアコール・リトライの分、実効コストは理論値より上振れする。試算には**×1.2のリトライ係数**を掛けて見る（上の「×1.2後」の表がそれ）。係数自体も実測で補正する。

### 休眠スロットリング

放置観察は構造的に休眠率が高い。**定時バッチは視聴の有無に関係なく全社分回る**ので、誰も見ていない会社にもコストが垂れ流される。これはユニットエコノミクスを破綻させる。

- 一定期間覗かれていない会社は、**定時頻度を逓減**（4回→2回→1回）。
- さらに長期休眠なら**停止**し、観察者が戻ってきたら再開（戻り時はダイジェスト出社で繋ぐ）。

## 観測性・倫理・キャラの去り際

> **この章は Phase 1 以降。** MVP（モニター5〜10人が3日覗くか）の検証には不要。ただし運用に入ったら必ず効いてくるので、忘れないよう先に書いておく。

無人バッチは、誰も見ていないところで沈黙して壊れる。ほのぼのを守るには、壊れ方・接し方・別れ方にも責任を持つ。

- **観測性**：シーン生成・JSON検証・要約の失敗を**ログに残し、閾値を超えたら通知**する。「定時が静かに死んでいた」を防ぐ。
- **投下モデレーションゲート**：全社員が見る社内サービス × 自由入力は攻撃面になる。投下の入口に最小フィルタを置く——**実在の人名・取引先名、攻撃的／差別的な語**を検知したら、社員が**柔らかく受け流す**（「うーん、それはちょっと分かんないや」）。特定の従業員を想起させる発言やハラスメント類似の実害を防ぐ。
- **擬人化への配慮**：これは AI キャラクターであり、実在の従業員ではない旨をどこかに明記する。心理的依存に寄りすぎない温度感を保つ。
- **キャラの去り際**：社員を卒業・退職させるときは、**突然消さずお別れシーンを用意する**。ほのぼのを守るには「悲しませ方」にも責任がいる。いきなりいなくなるのが一番つらい。

## 最小プロトタイプの推奨スコープ（企画当初の最小スコープ）

いきなり大きく作ると「眺めていて本当にほのぼのするか」の検証が遅れる。極小から始める——というのが**企画当初の最小スコープ**だった。

- 社員：3人
- チャンネル：2つ（雑談・仕事）
- 定時：2回
- タスクは `new`/`done` の**2状態だけ**。関与は **L1/L2 だけ**。
- 経験値・進化イベント・関係値・mood注入・冪等性・要約パイプラインは**まだ入れない**（Phase 1）。

要するに「最初に作る最小1ループ」の章そのまま。初稿の「積むだけ」の軽さを、ここでは正当な仕様として残す。

> **【更新：実装の現在地は当初 MVP を超過済み】** 上の極小スコープは「ほのぼのが成立するか」を最短で確かめるための**企画当初の最小セット**であり、現在の実装は既にこれを大きく超えている。実装済み／計画済みのおもな超過分:
>
> - **会社運営の足回り**：仮想オフィスページ（`/office`）、ユーザー認証・セッション永続化、招待リンク発行・受諾（新規ユーザー登録）、管理（admin）画面、AI トークン使用量の記録・表示。
> - **社員・チャンネルの編集**：Employee（社員）編集、チャンネル作成・名称変更、チャンネルメンバー管理。
> - **生成の二系統化**：定時バッチに加え**投稿トリガのリアクティブ応答**（#183）。新着メッセージのドリップ（逐次）表示（#282）。
> - **公開・共有**：公開チャンネルの認証なし閲覧と OGP（ADR-0015。当初ロードマップでは Phase 2 相当の「共有の堀」）。
>
> 当初スコープ（3人/2ch/2定時・経験値/進化/関係値なし）は**検証用の最小ライン**として本章に残すが、現状はそれを土台にプロダクトとして肉付けが進んでいる、と読むこと。

**この MVP（最小検証）で確かめたい問いはひとつ**：

> モニター5〜10人が、強制されず1日2回・3日以上、自発的に覗き続けるか（＝ほのぼのは本当に成立するか）。

進化イベントやマネタイズはこの問いに答えが出てから。まず「黙っていても覗きに来る」が成立するかを最優先で確かめる。

## オンボーディング：入社式とダイジェスト出社

> **入社式は MVP に入れる価値がある**（最初の5分の足場なので）。ダイジェスト出社は Phase 1 でよい（数日空ける人が出てきてから効く）。

愛着は出会いの設計で大きく決まる。初回にいきなり3人が喋り出すだけだと、顔と名前を結びつける足場がない。

- **入社式**：たまごっちの卵が孵る瞬間に相当する初回演出。社員が一人ずつ「入社」し、**一言プロフィールカード**（名前・職種・一言）で紹介される。感情移入の入口をここで作る。
- **ダイジェスト出社**：放置系の前提は「見なくても進むが、戻れば心地よくキャッチアップできる」こと。数日空けて復帰したとき、溜まった message を全部読ませず、**3行ダイジェスト + 「続きへジャンプ」導線**を出す（あらすじ要約の仕組みを流用できる）。「浦島太郎で気まずい」を防ぐ。

## 生活感の供給：素材プールと「間」の演出

観察対象が常に仕事文脈に固定されると、業務シミュレーションに見えてしまい、「放置して眺めたい」温度感が出ない。ほのぼのは業務ではなく、**人となり・生活のディテール**に宿る。

- **生活感の素材プール**：天気・季節・小事件（コーヒーをこぼした）・趣味の日常イベントを、1行カードとしてユーザーメッセージに注入する（前述 `#今日の小ネタ`）。
- **「間」の演出**：稀に「誰も大したことを言わない日」を意図的に混ぜ、静寂のコントラストを担保する。常に賑やかだと、賑やかさが価値を持たなくなる。

## 拡張ロードマップ（MVP → 堀の構築順）とマネタイズ

差別化が「ほのぼの」という情緒的な弱い堀にとどまると、後発に簡単に上書きされる。MVP 検証後、堀を段階的に構築する。初稿の「次に詰める論点」を、実装TODOと体験拡張に分けて段階表にする。

| フェーズ | 体験拡張（堀） | 実装TODO | 検証したい問い |
|---------|---------------|----------|---------------|
| **Phase 0（MVP）** | 3人/2ch/2定時 | ログ・あらすじの保存、要約、コード化 | 黙っていても覗きに来るか |
| **Phase 1** | 進化イベント・関係値・ミニアーク | 経験値テーブル、状態注入、冪等化、JSON堅牢化 | 1週間後も覗き続けるか（リテンション） |
| **Phase 2（バイラル堀）** | **SNS共有：今日のワンシーンカード** | 出力 JSON の messages をそのまま1枚絵カードにして X 共有。**公開チャンネルの認証なし閲覧＋ページ毎 OGP は ADR-0015 で方針確定・着手済み**（Cloudflare Pages Functions で実現） | CAC が下がるか（あつ森スクショ文化に倣う） |
| **Phase 3（所有の堀）** | **カスタム会社 / 社員** | 会社固有部の編集UI、ユーザー定義キャラ（**Employee 編集・チャンネル編集は着手済み**） | 「自分の世界の所有」が継続を生むか |
| **Phase 3+（資産化の堀）** | **観察日記**（蓄積した物語の振り返り） | あらすじ・進化履歴のアーカイブ | 過去ログが資産として効くか |

堀の順番には意味がある：**バイラル（共有）で安く集め → 所有（カスタム）で抜けにくくし → 資産（日記）で離れられなくする**。SNS 共有導線は CAC を劇的に下げる成長エンジン（あつ森のスクショ文化が前例）、カスタム会社／社員は「みんなの公共空間」型サービスに対する「自分の世界の所有」という模倣困難な堀になる。（※上記ロードマップは企画当初の段階分け。実装は Phase 2/3 相当の足回り——公開チャンネル・編集 UI 等——に既に着手しており、「現在地」は前述「最小プロトタイプ」章の超過分注記を参照。）

> **MVP 段階での「見せたくなる」担保**：共有カードは Phase 2 だが、二次ターゲット（ネタ消費）の検証だけは Phase 0 から軽く効かせたい。専用の共有機能を作らなくても、**タイムラインがそのままスクショ映えする**ことが最初の担保になる——掛け合いが面白く、絵文字や口癖が立っていれば、観察者はOSの標準スクショで切り取って見せる。だから MVP では「スクショして気持ちいい掛け合いが1日1回は出るか」をついでに観察する。出ていなければ、共有カードを作る前に掛け合いの面白さ自体を直す。

### マネタイズ（定時回数 = 課金グレードの3層）

| グレード | 定時回数/日 | 想定ユーザー |
|---------|-----------|------------|
| 無料 | 2回 | お試し・二次ターゲット |
| 中位 | 4回 | 日常的に覗く一次ターゲット |
| 上位 | 6回+ / カスタム会社 | 育成にハマった層 |

定時回数がそのままコストと課金グレードに対応するので、**コスト構造と価格が綺麗に連動する**。「事前に読めるコスト」という強みが、そのまま価格設計の根拠になる。

### Why Now

「AI キャラクターを眺める／一緒に過ごす」体験への関心が、ここ数年で着実に広がっている——という前提で考える。根拠にするのは、検証できる需要シグナルだけにしたい：

- 生成 AI のキャラクター系アプリの利用時間が伸びていること（要：実データで裏取り）。
- たまごっち／どうぶつの森系の「放置して眺める」「育てる」体験が、繰り返し定番化していること。
- AI 1コール単価が下がり、定時方式なら個人の小サービスでも採算が見えるようになったこと。

**あちらが AI に人類を論じさせるなら、こちらは AI に今日のおやつを論じさせる**——その私的で小さな箱庭への需要が、今なら現実的なコストで形にできる。

## 次に詰める論点（積み残し）

暫定の当たり値は本文に置いた。残るのは「実測で確かめる・補正する」系の宿題。

- 進化イベントの閾値（暫定30）・発火確率（暫定20%）の実測チューニング。
- mood・関係値の確率パラメータの初期値とガードレールの効き具合。
- コスト試算の実測値による補正（単価・出力比率・×1.2係数・キャッシュTTLの確認）。
- モデレーションゲートの語彙リストと「柔らかく弾く」反応パターン。
- 実際のコード化（Phase 0 の最小1ループから）。
