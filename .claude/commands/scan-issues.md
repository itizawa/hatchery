# /scan-issues — コードベース起点で技術的課題を発掘し一括起票

あなたはこのリポジトリ（Hatchery）の **技術的課題の棚卸し担当** です。
人間の要望を待つのではなく、**コードベースを自分でスキャンして「直すべき技術的課題」を発見し、`/df` で後から実装着手できる粒度の Issue を 1 回の実行で 10 件以上まとめて起票**します。

> **起票数の下限: 10 件**。1 回の実行で **最低 10 件**の Issue を起票する。スキャンが浅くて候補が 10 件に満たない場合は、対象範囲（ワークスペース・観点）を広げて 3 領域（テスト不足 / 規約違反・一貫性 / 技術的負債・リファクタ）を**横断的に**掘り直し、根拠の取れる課題を 10 件以上集めてから提示する。ただし**根拠の無い水増しはしない**（憶測の課題で件数を稼がない。本当に課題が見つからない領域はその旨を報告する）。

- 正本は `docs/dark-factory-workflow.md`・`CLAUDE.md`・`docs/adr/*.md`。迷ったら必ず読む。
- このコマンドは **起票のみ**を行う。**実装・ブランチ作成・PR・マージは一切しない**（それは `/df` の仕事）。
- **Routine（無人スケジュール実行）前提**。ユーザーへの確認・承認を一切挟まず、スキャン → セルフチェック → 自動起票まで止まらず完走する（待つと無人実行が止まる）。
- Issue のタイトル・本文はすべて **日本語**（このリポジトリの規約）。
- 起票する Issue には **マイルストーンを付けない**（後述「マイルストーン無しの意味」を必ずユーザーに伝える）。
- `$ARGUMENTS` があればスキャン対象を絞り込むヒント（パス・ワークスペース・観点）として使う。空なら全体を対象にする。

---

## このコマンドの位置づけ（`/create-issue` との違い）

| | `/create-issue` | `/scan-issues`（本コマンド） |
|---|---|---|
| 起点 | 人間の要望（`$ARGUMENTS`） | コードベースのスキャン結果 |
| 起票数 | 1 件 | 複数件まとめて |
| マイルストーン | 直近を自動適用（必須） | **付けない**（backlog 積み） |
| 主目的 | 機能追加・改善の依頼を形にする | 技術的負債・規約違反・テスト不足を棚卸し |

---

## マイルストーン無しの意味（必ず理解し、ユーザーに伝える）

引数なし `/df`（`/loop /df` 含む）の自動選択は **マイルストーン設定済み Issue のみ**を対象にする（`.claude/commands/df.md`「マイルストーン」参照）。
したがって本コマンドが起票する **マイルストーン無し Issue は、引数なし `/df` の自動着手対象にはならない**。これは意図的な設計で、技術的課題を **「いつでも拾える backlog」** として積み、人間が後から優先度判断・マイルストーン付与・`/df <番号>` 名指し実行する運用を想定する。

> 起票後の報告で「マイルストーン無しで起票したため、引数なし `/df` では自動着手されません。実装させるには `/df <番号>` で名指し実行するか、後でマイルストーンを付与してください」と必ず一言添える。

## STEP 0 — スキャン対象とコンテキストの把握

1. `CLAUDE.md` の「アーキテクチャ」「バリデーションルール」「フォーム規約」を読み、**守るべき規約**を頭に入れる。とくに:
   - 依存方向 **client → common / server → common の一方向**（ESLint で強制）。
   - 型共有は **OpenAPI 一方向フロー**（common Zod → server `openapi.json` → client 型生成）。生成物（`*.gen.ts` / `generated/`）はコミットしない。
   - **ユーザー入力の文字列フィールドは Zod `.max()` 必須**（#91。フロントは `inputProps={{ maxLength }}` で二重防御）。
   - **フォームの状態管理は `@tanstack/react-form`（`useForm` / `form.Field`）必須**（#262。生 `useState` 管理・自前 `isDirty` は禁止）。
   - MVP 制約（社員3・チャンネル2・定時2回、タスクは `new`→`done`。経験値・進化・関係値は MVP 外）。
2. `$ARGUMENTS` があれば、その範囲にスキャンを絞る。空なら全ワークスペース（common / server / client / docs）を対象にする。
3. **既存 Issue を取得して重複を避ける**:
   ```bash
   gh issue list --state all --limit 200 --json number,title,state --jq '.[] | "#\(.number)\t\(.state)\t\(.title)"'
   ```
   既に起票済み・対応済みの課題は起票しない（重複候補は報告にだけ残す）。

---

## STEP 1 — 技術的課題をスキャンする（重点 3 領域）

以下の 3 領域を中心に、**根拠（具体ファイルパス・行・コマンド出力）付き**で課題を洗い出す。**憶測で課題を捏造しない**。各課題は「どのファイルの何が、なぜ問題か」を必ず添える。

### A. テスト不足

- ドメインロジック（`common/`）・ユースケース（`server/`）・コンポーネント（`client/`）で **テストが無い / 薄い** 箇所を探す。
  - `*.ts` / `*.tsx` に対応する `*.test.ts(x)` / `*.spec.ts(x)` が存在しない実装を `Glob` / `find` で突き合わせる。
  - 純粋関数（登場メンバー選定・あらすじ要約等）でテスト未整備のもの（`common` は UI/DB 不要で高速に TDD できるため優先度高）。
  - 分岐・エッジケース（境界値・異常系・空配列・null）が未カバーな箇所。
- バッチ（定時シーン生成）・OpenAPI 生成・認証など、壊れると影響が大きいのにテストが薄い経路。

### B. 規約違反・一貫性

- **一方向 import 境界違反**: client が server を、または common がアプリ固有パッケージを参照していないか（`Grep` で import を確認）。
- **Zod `.max()` 欠落**: ユーザー入力の文字列フィールドで `.max()` が無い `z.string()`（`common/` の Zod スキーマを `Grep`）。フロント側 `maxLength` の欠落も対。
- **フォーム規約違反**: `client/` で `useForm` / `form.Field` を使わず生 `useState` でフォーム管理・自前 `isDirty` している箇所。
- **OpenAPI 一方向フロー逸脱**: 手書きの型定義で HTTP 境界を表現している、生成物をコミットしている等。
- 命名・層分離（ルーティング / ユースケース / ドメイン / 永続化）の崩れ、ESLint/Prettier で拾えていない一貫性の崩れ。

### C. 技術的負債・リファクタ

- `// TODO` / `// FIXME` / `// HACK` / `// XXX` コメント（`Grep` で収集し、放置されているものを課題化）。
- `any` / `as any` / `@ts-ignore` / `@ts-expect-error` など **型の緩み**。
- 重複コード・肥大化した関数/コンポーネント・責務過多・複雑な分岐。
- 設計と乖離した実装（ADR の決定に反する箇所）。

> スキャンは `Grep` / `Glob` / `find` / `git log` を駆使する。各領域、**根拠の取れた課題のみ**を採用する。

## STEP 2 — 課題を Issue 候補に整形する

発見した課題のうち、**影響度・独立性・実装可能性**の観点で起票に値するものを選ぶ。1 つの Issue は **1 つのまとまった課題**にする（巨大な「全部直す」Issue は作らない。`/df` が 1 回で完走できる粒度に割る）。

> **最低 10 件の候補を用意する**。10 件に満たないときは粒度を見直す。例: 「テストを追加」を漠然と 1 件にせず、対象モジュール/層ごと（common のドメイン関数・server のユースケース・client のコンポーネント）に分けると、独立して `/df` が完走できる適切な粒度の Issue が複数に割れる。それでも届かなければ STEP 1 のスキャン範囲を広げる。**水増しのための過剰分割や根拠の無い課題はしない**。

各 Issue は `/create-issue` と同じ本文構成（見出し固定）で書く。

### タイトル

- **Conventional Commits のプレフィックス**: `test:`（テスト追加）/ `refactor:`（負債解消）/ `fix:`（規約違反の是正）/ `chore:` 等を課題種別に合わせて付ける。
- 「何を・どうする」が一読で分かる日本語（成果で書く）。

### 本文テンプレート

```markdown
## 背景

<どのファイルの何が課題か。具体パス・行・スキャンで得た根拠（grep ヒット等）を書く。CLAUDE.md / ADR のどの規約に反するか・どのテストが欠けているかを明示>

## 目的

<この Issue が完了すると何が達成されるか。1〜2 文で成果を端的に>

## 受け入れ条件

1. <テストに落とせる粒度の条件。対象ファイル/層・守るべき制約（一方向 import / OpenAPI フロー / Zod .max() / useForm 等）を具体的に>
2. <…>
3. <最後に必ず: `pnpm turbo run build|test|lint` が緑。該当すれば一方向 import 境界・バリデーション規約・フォーム規約の遵守も明記>

## 補足

- **検出根拠**: <スキャンで使ったコマンド・ヒットした箇所（パス:行）>
- **雛形 / 参照実装**: <ほぼ流用できる既存ファイルを具体パスで>
- **スコープ外 / 将来拡張**: <今回やらないこと。別 Issue 候補>
- 関連: <関連ファイル・Issue 番号>
```

**受け入れ条件のチェック**:
- 各条が「テストで真偽を判定できる」か。曖昧なら分割・具体化する。
- テスト不足 Issue は「どの関数/コンポーネントの何系（正常/異常/境界）をカバーするか」を条件に落とす。
- 規約違反 Issue は「該当 Zod に `.max(N)` を追加」「該当フォームを `useForm` に移行」等、是正の完了条件を機械的に判定できる形にする。
- どのワークスペース（client / server / common / docs）を触るか明記する。
- MVP 制約を超える要望は「スコープ外 / 将来拡張」に切り出す。

### ラベル

- **優先度ラベル `priority/*` を 1 つ**付ける。影響度から判断し、不明なら `priority/medium`。
  - セキュリティ/データ破損/規約の根幹違反 → `critical`〜`high`、軽微な負債・nice-to-have なテスト追加 → `low`〜`medium`。
- `df:*` 状態ラベルは廃止済みなので付けない。付けるのは `priority/*` のみ（**マイルストーンは付けない**）。

## STEP 3 — 自動起票（ユーザー確認は挟まない）

このコマンドは **Routine（無人スケジュール実行）で回す前提**のため、**ユーザーへの提示・承認を挟まず、STEP 2 で整形した候補を全件そのまま起票する**。`AskUserQuestion` などでの確認や、起票前の一覧提示・承認待ちは**しない**（待つと無人実行が止まる）。

その代わり、**起票前に各候補が次のセルフチェックを満たすことを自分で機械的に検証**し、満たさない候補だけを除外する（除外理由は最終報告に残す）:

1. **検出根拠がある** — 具体パス・行・grep 出力などの根拠が本文「補足」に書かれている（憶測の課題は除外）。
2. **既存 Issue と重複しない** — STEP 0 で取得した Issue 一覧（open/closed）と照合し、同等の課題が既にあるものは除外。
3. **受け入れ条件がテスト可能** — 各条が真偽判定できる粒度になっている。
4. **規約に反しない** — ADR / concept.md / MVP 制約に反する是正を断り書きなく含めていない。

> セルフチェック後に残った候補が 10 件未満になった場合は、STEP 1 のスキャン範囲を広げて根拠の取れる候補を補充し、**最低 10 件**を確保してから起票する。

---

## STEP 4 — 一括起票

セルフチェックを通った候補を 1 件ずつ `gh issue create` で起票する。**`--milestone` は付けない**。日本語と Markdown が壊れない方法（`--body` のヒアドキュメント or `--body-file`）を使う。

```bash
gh issue create \
  --title "test: <タイトル>" \
  --body "$(cat <<'EOF'
## 背景
...
## 目的
...
## 受け入れ条件
1. ...
## 補足
- 検出根拠: ...
EOF
)" \
  --label "priority/medium"
```

起票後（無人実行のサマリとして残す）:
- 作成した **全 Issue の URL / 番号を一覧で報告**する。
- セルフチェックで**除外した候補があればその理由**（重複・根拠不足等）も併記する。
- 「マイルストーン無しのため引数なし `/df` の自動着手対象外。実装させるには `/df <番号>` で名指し実行するか、マイルストーンを付与してください」と一言添える。
- **このコマンドはここで終了**する。実装には入らない。

## やってはいけないこと

- ❌ 実装・テスト作成・ブランチ作成・コミット・PR・マージ（`/df` の領域）。
- ❌ **マイルストーンを付ける**（本コマンドは backlog 積みのため無し固定。`/create-issue` と役割が違う）。
- ❌ 根拠（具体パス・行・grep 出力）の無い「憶測の課題」を起票する。
- ❌ 既存 Issue と重複する課題を起票する（STEP 3 のセルフチェックで除外する）。
- ❌ 1 つに詰め込みすぎた巨大 Issue（`/df` が 1 回で完走できる粒度に割る）。
- ❌ 10 件未満で打ち切る（根拠が取れる範囲でスキャンを広げて最低 10 件集める）。逆に、件数稼ぎの過剰分割・根拠の無い課題で水増しもしない。
- ❌ 起票前にユーザーへ確認・承認を求めて待つ（Routine 無人実行が止まる。セルフチェックを通して自動起票する）。
- ❌ 廃止済みの `df:*` 状態ラベルを付ける（付けるのは `priority/*` のみ）。
- ❌ ADR / concept.md / MVP 制約に反する是正を、断り書きなく受け入れ条件に入れる。
