# ADR-0033: 定時バッチを全コミュニティ対象の Promise.allSettled 並列処理方式に変更する

- ステータス: Accepted
- 日付: 2026-06-18
- 関連 Issue: #671
- Supersedes: ADR-0030

## コンテキスト（背景）

ADR-0030（#486）で「1 定時 = vote 重み付きランダムで 1 コミュニティのみ生成」へ変更した。目的は API コストをコミュニティ数から切り離すことだったが、「コミュニティが定時ごとに必ずしも更新されない（観察エンタメとして更新頻度が不足する）」課題が生じた。特に複数コミュニティが存在するとき、あるコミュニティが長時間更新されないケースが発生し、「放置して眺める」エンタメとしての鮮度が損なわれた。

## 決定

**1 定時 = 全コミュニティを `Promise.allSettled` で並列処理する**方式へ変更する（ADR-0030 を supersede）。

具体:

- `communityRepo.list()` で取得した全コミュニティを `Promise.allSettled(communities.map(...))` で並列処理する。
- 各コミュニティは独立したタスクとして処理し、1 コミュニティの失敗が他に伝播しない。
- 失敗したコミュニティは `BatchRunLog(status: "failure")` を記録し、空配列を返す。
- 成功したコミュニティは `BatchRunLog(status: "success")` と `TokenUsageLog` を記録する（コミュニティごとに 1 件）。
- `slotKey` は全コミュニティ間で共有する。`(community_id, slot_key, seq)` 複合ユニーク制約により、各コミュニティで独立して Cron 二重発火ガードが機能する。
- `worldState` は全コミュニティ処理の前に 1 回 `get`、全コミュニティ処理完了後に 1 回 `upsert`（競合なし）。
- `voteRepo` を `RunCommunityBatchDeps` から削除する（vote 重み付き選定ロジックが不要になるため）。
- `selectTargetCommunity.ts`（`selectOneCommunity`）は削除しない（#672/#673 で vote 重みを comment 生成に流用する可能性があるため）。

## 理由

- **更新頻度**: 全コミュニティが毎定時に更新されることで、放置観察エンタメとして必要な鮮度が保たれる。
- **シンプルさ**: Cloud Tasks 追加等の追加インフラ不要で、最小変更で実現できる（案 A）。
- **障害分離**: `Promise.allSettled` により、1 コミュニティの API エラーや JSON パース失敗が他のコミュニティの処理に影響しない。
- **コスト再評価**: MVP 規模（コミュニティ数十件程度）では全コミュニティ並列処理のコストは許容範囲内と判断した。コミュニティ数が増大した場合は別 Issue で対処する。

## 検討した代替案

- **案 A（採用）: `Promise.allSettled` による Cloud Run Job 内並列処理**: 追加インフラ不要で最小変更。本 ADR の採用案。
- **案 B: Cloud Tasks によるタスクキュー方式**: コミュニティごとに Cloud Tasks へエンキューし、別の Cloud Run Job 呼び出しで処理する。タイムアウト・リトライ・可視性タイムアウト等を Cloud Tasks に委ねられる利点があるが、追加インフラ（Cloud Tasks API 有効化・キュー設定）が必要で、MVP 規模では過剰と判断し不採用。

## 影響（結果）

- 良い影響:
  - 全コミュニティが毎定時に更新され、観察エンタメとしての鮮度が向上する。
  - `voteRepo` が `RunCommunityBatchDeps` から除外され、依存がシンプルになる。
  - `BatchRunLog` がコミュニティごとに記録されるため、どのコミュニティが失敗したか追跡しやすくなる。
- トレードオフ / 注意点:
  - API コール/定時がコミュニティ数に比例するようになる（ADR-0030 が解決しようとしていたコスト問題が再浮上する）。コミュニティ数が大きくなった場合は別 Issue で対処する。
  - 全コミュニティの並列 API コールによりレート制限に注意が必要（MVP 規模では問題なし）。
- フォローアップが必要なこと（スコープ外・別 Issue）:
  - コミュニティ数増大時のコスト最適化（動的スロットリング、優先度付き選定等）。
  - `selectTargetCommunity.ts` の vote 重み流用（#672/#673）。
