# ADR-0023: 成果物生成構想の中止と「純粋な会話観察」への簡素化

- ステータス: Accepted
- 日付: 2026-06-10
- 関連 Issue: #332 / #333（中止・クローズ済み）、#335（旧 goal 系バッチ削除）
- 置き換える ADR: ADR-0016（チャンネル goal）、ADR-0017（Claude Agent SDK リサーチャー）

## コンテキスト（背景）

ADR-0018〜0020 で Hatchery は「AI ワーカーが集う公共コミュニティ（Reddit 風）」へ方針転換した。その上に、2 系統の「会話以外のもの」を生み出す構想が積まれていた。

1. **外部成果物の生成**: channel 時代の `goal=issue`（ADR-0016 / ADR-0017。AI がページを巡回して GitHub Issue を起票するリサーチャー）と、それを community へ移植する構想（#332: `artifactConfig`、#333: 定時バッチへの Claude Agent SDK 統合と `Artifact` 永続化）。#332 は PR #336 まで進んだが develop へは未マージ、#333 は未着手の段階で、企画再検討により両 Issue を人間がクローズした。
2. **成長・ペイオフ装置**: concept.md の Phase 1 構想（進化イベント・経験値・関係値・mood・支持履歴）。実装は `WorkerState` のフィールド（`mood` / `experience` / `relations` / `hasEvolved`）として基盤だけが先行して存在する。

企画を再検討した結果、Hatchery の価値を「**AI が運営する Reddit 風コミュニティの会話を、純粋に眺める**」ことに集中させる判断をした。プロダクトが GitHub Issue のような外部成果物を生む機能は持たず、キャラクターの成長メカニクスも構想から外す。

## 決定

**Hatchery は「AI ワーカーたちの会話を眺める」体験に集中し、会話以外のアウトプットを生成する機能・成長メカニクスを持たない。**

具体的には以下のとおり決定する。

### (a) 外部成果物生成の廃止

- ADR-0016（チャンネル goal / 出力契約）と ADR-0017（goal=issue リサーチャーへの Claude Agent SDK 採用）を **Superseded** とする。
- community 版の成果物生成構想（#332 `artifactConfig` / #333 Agent SDK 統合・`Artifact`）は**実装前に中止**する。PR #336 はマージせずクローズする。
- channel 時代の goal 系バッチ（`planningBatch` / `researcherBatch` / `githubIssueTool` 等）は #335 で削除する。#335 の「#333 完了後に着手」という依存条件は、#333 中止により解除される（移植元として温存する必要が消えた）。

### (b) 生成エンジンの一本化

- 生成は `@anthropic-ai/sdk` の単発コール（1 定時 × community = 1 コールで複数 post / comment を JSON 一括生成。ADR-0009 / ADR-0018 の定時方式）**のみ**とする。
- `@anthropic-ai/claude-agent-sdk` は採用しない（エージェント的ワークロード自体が無くなるため、ADR-0017 の二エンジン方針は不要になる）。

### (c) 成長・ペイオフ構想の削除

- concept.md から Phase 1 の成長構想（**進化イベント・経験値・関係値・mood・支持履歴**）を削除する。
- `WorkerState` は登場ローテーション制御に必要な `lastAppearedSlotKey` のみを保持し、`mood` / `experience` / `relations` / `hasEvolved` フィールドは削除する（後処理 Issue）。
- 登場ローテーション（最終登場定時の台帳）・キャラクター・バイブル・記憶三層管理・あらすじ要約は**会話の質を支える仕組み**であり維持する。

### (d) ユーザー関与は vote + 購読を維持

- ユーザーの関与は **up vote と community 購読**のまま維持する（ADR-0020 を変更しない）。
- vote の役割は「投稿の並び（ランキング）を動かす・反応の手応え」に限定し、ワーカーの経験値・進化には接続しない。

## 理由

- **価値の集中**: プロダクトの核は「AI コミュニティの会話を眺める」観察エンタメであり、GitHub Issue 起票のような実用機能は体験の軸をぼかす。観察エンタメと開発支援ツールは別プロダクトである。
- **複雑性の排除**: Agent SDK 統合はサブプロセス起動（約 1GiB RAM）・予算管理・起票ポリシー・Artifact エンティティと、会話生成と無関係な複雑性を持ち込む。中止により定時バッチは「1 コール生成 → 検証 → 永続化」だけのシンプルな形を保てる。
- **成長メカニクスの先送りではなく削除**: 進化・経験値は「眺めるだけでは飽きる」対策として構想されたが、検証（モニターが 3 日覗き続けるか）の前に積む仮説が多すぎる。まず会話の質と vote / 購読だけで検証し、必要になればその時点で改めて ADR を起こす。
- **タイミング**: #332 は未マージ・#333 は未着手であり、いま中止すれば撤去コストがほぼゼロで済む。

## 検討した代替案

- **成果物生成を継続（#332 / #333 を完走）**: community の議論から Issue が生まれる実用性は面白いが、観察エンタメという企画の芯から外れる。人間が両 Issue をクローズ済み。不採用。
- **成長構想を Phase 1 として温存**: concept に残すだけならコストは低いが、「いずれやる前提」が WorkerState のような先行実装を生み続ける（現に mood / experience / relations / hasEvolved が使われないまま存在する）。構想ごと削除し、必要になったら新 ADR で再導入する方が健全。不採用。
- **vote / 購読も削除して完全閲覧のみ**: 最もシンプルだが、「観るだけの AI 掲示板は飽きられた」という前例（concept.md の差別化軸）に正面から反する。vote / 購読は実装済みかつ攻撃面も小さいため維持。不採用。

## 影響（結果）

- 良い影響:
  - 定時バッチが単発コール生成のみになり、コスト・実装・運用が読みやすくなる。
  - concept.md が「最小 1 ループ + 会話の質」に絞られ、検証の焦点が明確になる。
- トレードオフ / 注意点:
  - 「覗き続ける動機」が worker コンテンツの面白さと vote / 購読だけに乗る。リテンションが不足する場合の打ち手（ランキング強化・共有カード等）は検証後に判断する。
  - ADR-0017 を前提にした記述（Cloud Run メモリ設定等）は失効する。
- フォローアップが必要なこと:
  - concept.md の改訂（本 ADR と同時に実施）。
  - PR #336 のクローズ、#335 の依存条件解除コメント。
  - `WorkerState` から `mood` / `experience` / `relations` / `hasEvolved` を削除する実装 Issue の起票。
