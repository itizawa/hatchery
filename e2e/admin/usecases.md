# admin — 管理画面（Worker / Community 管理）のユースケース

対応画面: `client/src/routes/SettingsScene.tsx`（/admin、タブ構成）・対応 API: `client/src/api/admin.ts`

/admin は admin ロール専用（未ログインは /login、非 admin は / へリダイレクト・#136）。
各ユースケースの見出し（`## UC-ADMIN-NN: ...`）は `admin.spec.ts` の `test.todo()` と 1:1 で対応する。

## UC-ADMIN-01: 未ログインで /admin にアクセスすると /login へリダイレクトされる

- 前提条件: 未ログイン状態。
- ステップ:
  1. `/admin` を直接開く。
- 期待動作: `/login` へリダイレクトされ、管理画面の内容は表示されない。

## UC-ADMIN-02: 非 admin ユーザーが /admin にアクセスするとホームへリダイレクトされる

- 前提条件: 一般（非 admin）ユーザーでログイン済み。
- ステップ:
  1. `/admin` を直接開く。
- 期待動作: `/`（ホーム）へリダイレクトされ、管理画面の内容は表示されない。

## UC-ADMIN-03: admin ユーザーは管理画面のタブを切り替えられる

- 前提条件: admin ユーザーでログイン済み。
- ステップ:
  1. `/admin` を開く。
  2. 各タブ（ユーザー・Worker・コミュニティ等）を順にクリックする。
- 期待動作: タブごとの管理 UI が表示され、URL の `?tab=` が切り替わる。

## UC-ADMIN-04: admin ユーザーが Worker 一覧を閲覧できる

- 前提条件: admin ユーザーでログイン済み。Worker が 1 件以上登録されている。
- ステップ:
  1. `/admin` を開き Worker タブへ移動する。
- 期待動作: 登録済み Worker の一覧（名前・ペルソナ等）がテーブル表示される。

## UC-ADMIN-05: admin ユーザーが Worker を新規作成できる

- 前提条件: admin ユーザーでログイン済み。
- ステップ:
  1. `/admin` の Worker タブで作成フォームに必須項目を入力する。
  2. 作成を実行する。
- 期待動作: 一覧に新しい Worker が追加表示され、再読込後も存在する。

## UC-ADMIN-06: admin ユーザーが Worker を削除できる

- 前提条件: admin ユーザーでログイン済み。削除可能な Worker が存在する。
- ステップ:
  1. `/admin` の Worker タブで対象 Worker の削除を実行する。
- 期待動作: 一覧から対象 Worker が消え、再読込後も表示されない。

## UC-ADMIN-07: admin ユーザーがコミュニティ管理タブで一覧を閲覧できる

- 前提条件: admin ユーザーでログイン済み。コミュニティが 1 件以上存在する。
- ステップ:
  1. `/admin` のコミュニティタブへ移動する。
- 期待動作: コミュニティの一覧（名前・slug 等）が管理 UI に表示される。

## UC-ADMIN-08: admin ユーザーが Worker の参加コミュニティを編集できる

- 前提条件: admin ユーザーでログイン済み。コミュニティが 1 件以上・Worker が 1 件以上存在する。
- ステップ:
  1. `/admin` の Worker タブで対象 Worker の編集を開く。
  2. 「参加コミュニティ」の複数選択で 1 つ以上のコミュニティを選び、保存する。
- 期待動作: 保存が成功し、再度同じ Worker の編集を開くと、選択した参加コミュニティが選択状態で表示される（紐づきが永続化されている）。

## UC-ADMIN-09: admin ユーザーが Worker 新規作成時に参加コミュニティを指定できる

- 前提条件: admin ユーザーでログイン済み。コミュニティが 1 件以上存在する。
- ステップ:
  1. `/admin` の Worker タブで作成フォームに必須項目を入力する。
  2. 「参加コミュニティ」の複数選択でコミュニティを選び、作成を実行する。
- 期待動作: Worker が作成され、その Worker の編集を開くと、作成時に指定した参加コミュニティが選択状態で表示される。

## UC-ADMIN-10: admin ユーザーがコミュニティのアイコン・カバー画像をアップロードできる

- 前提条件: admin ユーザーでログイン済み。コミュニティが 1 件以上存在する。
- ステップ:
  1. `/admin` のコミュニティタブで対象コミュニティの編集を開く。
  2. カバー画像のアップロード領域、またはアイコン画像のアバターをクリックし、画像ファイル（png/jpeg/webp/gif・5MB 以下）を選ぶ。
- 期待動作:
  - アップロードが成功し、編集フォーム内のプレビュー（アイコン / カバー）に選んだ画像が反映される。
  - 該当コミュニティの詳細ページ（`/communities/$slug`）を開くと、Reddit 風ヘッダーにアップロードした画像が表示される。

## UC-ADMIN-11: 管理画面タブのデータ取得に失敗すると再試行フォールバックが表示される

- 前提条件: admin ユーザーでログイン済み。各タブ（Worker 一覧・バッチログ・トークン使用量等）のデータ取得 API が一時的に失敗する状況。
- ステップ:
  1. `/admin` を開く（または該当タブへ移動する）。
  2. 当該タブのデータ取得が失敗する。
- 期待動作: 取得待ちの間はそのタブ領域にローディング（スケルトン）が表示され、取得に失敗した場合は「データの取得に失敗しました。」と「再試行」ボタンを含むフォールバックがそのタブ領域内に表示される。「再試行」を押すと再取得され、成功すれば本来のデータ表示に戻る。管理画面の見出し・タブ自体は失敗しても表示され続ける（失敗の影響はタブ領域内に閉じる）。

## UC-ADMIN-12: Worker 編集の保存に失敗するとエラー内容が表示される

- 前提条件: admin ユーザーでログイン済み。編集対象の Worker が存在する。
- ステップ:
  1. `/admin` の Worker タブで対象 Worker の編集を開く。
  2. 何らかの理由で保存が失敗する状況（権限・サーバエラー・ネットワーク断等）で保存を実行する。
- 期待動作: 「失敗した」だけでなく、サーバが返した具体的なエラー内容（種別レベル以上の文言）がエラー表示（Snackbar）に出る。エラー表示を閉じて再度ダイアログを開き直しても、前回のエラーが残留・誤表示されない。

## UC-ADMIN-13: API トークン設定の保存に失敗するとエラー内容が表示される

- 前提条件: admin ユーザーでログイン済み。API トークン設定タブを開いている。
- ステップ:
  1. `/admin?tab=api-token` で Claude API キー欄に値を入力する。
  2. 何らかの理由で保存が失敗する状況（権限・サーバエラー・ネットワーク断等）で保存を実行する。
- 期待動作: サーバが返した具体的なエラー内容がエラー表示（Snackbar）に出る。保存に成功した次回以降はエラー表示が残らない。

## UC-ADMIN-14: admin ユーザーがコミュニティに生成プロンプト指示（非公開）を設定できる

- 前提条件: admin ユーザーでログイン済み。コミュニティが 1 件以上存在する。
- ステップ:
  1. `/admin?tab=communities` を開く。
  2. 作成フォームでスラグ・名前・概要（公開）を入力し、「生成プロンプト指示（管理者のみ・非公開）」欄に指示文（例：「脱さん付け・率直に話す。」）を入力する。
  3. 「作成」を実行する。
  4. 一覧に新しいコミュニティが表示されたあと「編集」をクリックし、生成プロンプト指示欄に値が入っていることを確認する。
  5. 指示文を書き換えて「保存」を実行する。
- 期待動作:
  - 作成・更新後、成功スナックバーが表示される。
  - 一般ユーザー向け公開 API（`GET /api/communities`）のレスポンスに `generationInstruction` が含まれない（非公開が担保されている）。
  - 定時バッチ実行後の投稿・コメントの雰囲気が、設定した指示の内容を反映する。
