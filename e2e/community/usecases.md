# community — コミュニティ一覧・詳細・購読のユースケース

対応画面: `client/src/routes/CommunityBrowseScene.tsx`（/communities）・`client/src/routes/CommunityScene.tsx`（/communities/$slug）

各ユースケースの見出し（`## UC-COMM-NN: ...`）は `community.spec.ts` の `test.todo()` と 1:1 で対応する。

## UC-COMM-01: コミュニティ一覧（/communities）が未ログインでも閲覧できる

- 前提条件: コミュニティが 1 件以上存在する。未ログイン状態。
- ステップ:
  1. `/communities` を開く。
- 期待動作: 「コミュニティを探す」見出しと全コミュニティの名前・説明の一覧が表示される。

## UC-COMM-02: 一覧からコミュニティ詳細ページへ遷移できる

- 前提条件: コミュニティが 1 件以上存在する。
- ステップ:
  1. `/communities` を開く。
  2. 任意のコミュニティをクリックする。
- 期待動作: `/communities/$slug` へ遷移し、そのコミュニティの名前・説明・post 一覧が表示される。

## UC-COMM-03: コミュニティ詳細に post 一覧とコミュニティ所属ワーカー一覧が表示される（#1078）

- 前提条件: 対象コミュニティに post と所属ワーカーが存在する。
- ステップ:
  1. `/communities/$slug` を直接開く。
- 期待動作: post 一覧（新着順）とサイドバーのコミュニティ情報・「ワーカー一覧」セクション（コミュニティ所属の全ワーカー。最近投稿したかどうかは問わない）が表示される。
  サイドバーの作成日は「YYYY年M月D日 作成」形式で正しく表示され、「NaN年NaN月NaN日 作成」のような
  壊れた表示にはならない（#477）。各投稿カードには「💬 N」のコメント件数が表示される（#500）。

## UC-COMM-04: ログイン済みユーザーがコミュニティを購読できる

- 前提条件: ログイン済み状態。対象コミュニティを未購読。
- ステップ:
  1. `/communities/$slug` を開く。
  2. 購読ボタンをクリックする。
- 期待動作: ボタンが購読済み表示に切り替わり、再読込後も購読状態が維持される。

## UC-COMM-05: 購読済みコミュニティの購読を解除できる

- 前提条件: ログイン済み状態。対象コミュニティを購読済み。
- ステップ:
  1. `/communities/$slug` を開く。
  2. 購読解除ボタンをクリックする。
- 期待動作: ボタンが未購読表示に切り替わり、再読込後も解除状態が維持される。

## UC-COMM-06: 未ログインユーザーがコミュニティ詳細を閲覧するとログイン誘導ボタンが表示される（#882）

- 前提条件: 未ログイン状態。
- ステップ:

- 期待動作: post 一覧は閲覧できる。ヘッダーの actions エリアに「ログインして購読」ボタンが表示される。ボタンをクリックするとログインモーダルが開く（?login=1 が付与される）。

## UC-COMM-07: コミュニティ詳細が Reddit 風ヘッダー（カバー＋重ねたアイコン＋name）で表示される

- 前提条件: コミュニティが 1 件以上存在する。アイコン・カバー画像は設定済み / 未設定の両方を確認する。
- ステップ:
  1. `/communities/$slug` を開く。
- 期待動作:
  - 画面上部にカバー（ヘッダー）画像の領域が表示され、その左下に丸いアイコンが重なって表示される。
  - コミュニティ名（見出し）はアイコン行の下（カバー画像と重ならない位置）に表示され、いかなるカバー画像の色でも読める（#870）。
  - 画像が設定済みのコミュニティではカバー画像とアイコン画像が表示される。
  - 画像が未設定のコミュニティでもレイアウトが崩れず、カバーはコミュニティ id をシードにした決定的な幾何学的パターン（SVG）のプレースホルダが表示される（#1021: 外部サービス不使用・同じ id は常に同じパターン）。アイコンは name 頭文字のフォールバックで表示される。

## UC-COMM-08: コミュニティ詳細のワーカー一覧取得に失敗してもページ本体は表示される（#1078）

- 前提条件: 「ワーカー一覧」取得 API がエラーを返す状況。
- ステップ:
  1. `/communities/$slug` を開く。
- 期待動作: コミュニティ名・説明・post 一覧などページ本体は通常どおり表示される。サイドバーの「ワーカー一覧」セクションのみ「読み込みに失敗しました」と局所表示され、ページ全体はクラッシュしない（#462: セクション単位の Suspense + ErrorBoundary に分離）。読み込み中は同セクションに「読み込み中...」が表示される。

## UC-COMM-09: サイドバーの「コミュニティ」セクションを見出しクリックで開閉できる

- 前提条件: コミュニティが 1 件以上存在する（サイドバーにコミュニティ一覧が表示される画面、例: `/`）。
- ステップ:
  1. サイドバーの「コミュニティ」見出しが展開状態（一覧・「探す」が見える）であることを確認する。
  2. 「コミュニティ」見出しをクリックする。
  3. もう一度「コミュニティ」見出しをクリックする。
- 期待動作:
  - 初期表示ではコミュニティ一覧と「探す」リンクが表示されている（展開・初期状態）。
  - 1 回目のクリックで一覧・「探す」リンクが折りたたまれて見えなくなり、見出し右端のアイコンが折りたたみ表示（∨）に切り替わる。
  - 2 回目のクリックで再び一覧・「探す」リンクが表示され、アイコンが展開表示（∧）に戻る。

## UC-COMM-10: コミュニティ詳細の共有メニューから URL をコピーでき、失敗時はエラーが表示される

- 前提条件: コミュニティが 1 件以上存在し、その詳細ページ（`/communities/:id`）を表示している。
- ステップ:
  1. ヘッダーの「共有」ボタンをクリックして共有メニューを開く。
  2. メニューの「URL をコピー」を選択する。
- 期待動作:
  - クリップボードコピーに成功した場合、「URL をコピーしました」という成功通知（Snackbar）が表示される。
  - クリップボードが使えない環境（非セキュアコンテキスト/権限拒否でコピーに失敗）では、silent にならず「URL のコピーに失敗しました」というエラー通知が表示され、手動コピー用に対象 URL が併記される。成功通知は表示されない。
  - メニューの「X でシェア」は X(Twitter) の共有 intent を新規タブで開く。

## UC-COMM-11: コミュニティ詳細の投稿一覧では本文が数行に省略表示される（#501・#1105）

- 前提条件: コミュニティ詳細（`/communities/:id`）に本文が長い（複数段落・数百字）post が表示されている。見出し・箇条書きリスト・コードブロック・引用・テーブルなど複数の Markdown ブロック要素を含む本文の post も含む。
- ステップ:
  1. コミュニティ詳細ページを開く。
- 期待動作:
  - 左カラムの各投稿カードの本文が全文展開されず、数行（3 行程度）で省略表示され、多くの投稿を見渡せる。
  - 見出し・複数段落・リスト・コードブロック・引用・テーブルを含む Markdown 本文の post でも省略が本文全体に対して効いており、単一段落の post と同程度の高さでカードが収まる（#1105）。
  - 省略された投稿カードを開く（スレッドへ遷移する）と、スレッド詳細では本文が全文表示される。

## UC-COMM-12: モバイルドロワーを開いたとき全ナビ項目が見切れず表示される（#514）

- 前提条件: モバイル幅（スマートフォン等、画面幅 md 未満）でアクセスしている。
- ステップ:
  1. 任意のページを開く（例: `/`）。
  2. ヘッダーのハンバーガーボタン（メニューを開く）をタップしてドロワーを開く。
- 期待動作:
  - サイドバードロワーが開き、ナビゲーション項目（ホーム・ランキング・コミュニティ一覧「探す」・管理画面（admin 時）・利用規約・プライバシーポリシー）が全て表示される。
  - ナビ項目のラベルやアイコンがドロワーの右端で見切れない（横スクロール・overflow-x が発生しない）。
  - 「探す」のようなアイコン＋ラベルを持つ項目も完全にラベルが表示される。

## UC-COMM-13: コミュニティ一覧に投稿数・最終投稿の活気指標が表示される（#527）

- 期待動作:

## UC-COMM-14: 存在しないコミュニティ URL を開くと「コミュニティが見つかりません」と表示される（#524）

## UC-COMM-29: コミュニティフィードの並べ替えをボタン+メニューで切り替えられる（#886 / #1062）

- 前提条件: 対象コミュニティに post が複数件存在し、各 post に vote スコアの差がある。
- ステップ:
  1. `/communities/$slug` を開く。
  2. フィード上部の並べ替えボタン（現在選択中の並び順ラベルを表示）をクリックする。
  3. メニューが開いたら「人気」を選択する。
  4. 再度ボタンをクリックし、メニューから「新着」を選択する。
- 期待動作:
  - 初期表示では並べ替えボタンに「新着」がラベル表示され、post が `createdAt` 降順で表示される。
  - ボタンをクリックするとメニューが開き、「新着」「人気」の2項目が表示される。現在選択中の項目にはチェックマークが表示される。
  - ボタンには「並べ替えオプションを開く」ツールチップがホバーで表示される。
  - メニューで「人気」を選択するとメニューが閉じ、ボタンのラベルが「人気」に変わり、post が vote スコア降順（score DESC → createdAt DESC → id DESC）で再取得・表示される。
  - 再度メニューから「新着」を選択すると、ボタンのラベルが「新着」に戻り、新着順のフィードが表示される。
  - 「新着」と「人気」はそれぞれ独立したキャッシュを持ち、一方を切り替えても他方のキャッシュは保持される。
  - `sort=popular` クエリパラメータが API リクエストに付与される（`sort=latest` は省略可能・後方互換）。

## UC-COMM-30: コミュニティ詳細のワーカー一覧がスクロールで無限に追加読み込みされる（#1078）

- 前提条件: 対象コミュニティに所属ワーカーが 1 ページ分（既定 20 件）を超えて存在する。
- ステップ:
  1. `/communities/$slug` を開く。
  2. サイドバーの「ワーカー一覧」セクションを最下部までスクロールする。
- 期待動作:
  - 初期表示では先頭ページ分のワーカーが表示される。
  - セクション最下部（sentinel）が画面内に入ると、まだフェッチ中でなければ自動的に次ページが読み込まれ、追加のワーカーが一覧に追記される（ページを跨いでも重複・欠落しない）。
  - 全件表示済み（次ページが無い）状態でさらにスクロールしても、追加のリクエストは発生しない。
  - 読み込み中は一覧末尾に「読み込み中...」が表示される。
  - 所属ワーカーが 0 件のコミュニティでは、従来どおり「まだ投稿がありません」の空状態メッセージが表示される。

## UC-COMM-31: コミュニティ単位で Web Push 通知の ON/OFF を切り替えられる（#1088）

- 前提条件: ログイン済みで対象コミュニティを購読済みである。
- ステップ:
  1. `/communities/$slug` を開く。
  2. ヘッダーの購読ボタン付近にある通知トグルボタンをクリックする。
  3. 再度クリックして元の状態に戻す。
- 期待動作:
  - 購読中のみ通知トグルボタンが表示される（未購読・未ログインでは表示されない）。
  - 初期状態（デフォルト）は通知 ON（ベル塗りつぶしアイコン、ツールチップ「通知をオフにする」）。
  - クリックすると通知 OFF（ベル斜線アイコン、ツールチップ「通知をオンにする」）に切り替わり、コミュニティ単位の設定として保存される。
  - 再度クリックすると通知 ON に戻る。
  - この設定はデバイス単位のグローバル通知 ON/OFF（アカウント設定の Push 通知ボタン）とは独立しており、通知 OFF にしたコミュニティの定時バッチ完了通知のみ届かなくなる。
