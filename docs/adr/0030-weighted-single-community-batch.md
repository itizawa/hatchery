# ADR-0030: 定時バッチを vote 重み付きランダムで 1 コミュニティのみ生成する方式へ変更する

- ステータス: Accepted
- 日付: 2026-06-13
- 関連 Issue: #486

## コンテキスト（背景）

ADR-0009 / ADR-0018 / ADR-0019 で定めた定時方式は「1 定時で **全 community** を取得し、community ごとに 1 API コールで会話を生成する」ものだった。実装は `server/src/batch/runCommunityBatch.ts` で `communityRepo.list()` → 全 community ループ。

この方式では Claude API コストが **コミュニティ数 × 定時回数/日** に比例し、コミュニティを増やすほどコストが線形に増える。MVP（community 2 つ）を超えてコミュニティを拡張すると API コストが青天井になる構造的問題があった。

放置観察エンタメ（concept.md / ADR-0018）として理想的なのは「worker が定時ごとに、どれかのコミュニティを選んで会話する」体験であり、「今日どのコミュニティが動くか」自体が観察の楽しみになる。

## 決定

**1 定時 = vote 重み付きランダムで選んだ 1 コミュニティだけを生成・永続化する**方式へ変更する（ADR-0009 / ADR-0019 の「毎定時に全 community 生成」を一部 supersede）。

具体:

- 1 定時の Claude API コールは **常に最大 1 回**（コミュニティ数に非依存）。
- 選定は **vote 重み付きランダム**: エンゲージメント（vote を集めたコミュニティ）に報いる。
  - 重み ＝ そのコミュニティ配下の post / comment が集めた vote の**純スコア（up − down）の合計**。
  - 集計対象は**直近 7 日**（`Vote.createdAt` 基準）。期間は名前付き定数 `VOTE_WEIGHT_WINDOW_DAYS` として切り出し後から調整可能にする。
  - **cold start 床**: `weight = max(0, 純vote) + 1`。vote 0・新規コミュニティも必ず正の重みを持ち、稀に選ばれる（床が無いと新規が永久に選ばれない）。
- 責務分割（import 境界 server → common を維持）:
  - **重み付きランダム選定の純粋ロジックは common**（`common/src/logic/selectWeightedCommunity.ts`、乱数源注入で決定的にテスト）。
  - **vote の community 別純スコア集計は server**（`VoteRepository.netScoresByCommunitySince`。`Vote.targetId` は Post / Comment の多態参照なので DB join が必要）。
- community が 0 件のときは何も生成せず正常終了する（既存のスキップ挙動と整合）。
- `(community_id, slot_key, seq)` 複合ユニーク制約・slotKey による Cron 二重発火ガードは従来どおり機能する。

## 理由

- **コスト**: API コール/定時が固定（最大 1 回）になり、コストがコミュニティ数から切り離される。これがこの変更の主目的。
- **エンタメ性**: 各コミュニティの更新は間欠的になるが、「今日どのコミュニティが動くか」が観察の楽しみになり、放置観察エンタメの体験と整合する。
- **エンゲージメント還元**: vote を多く集めたコミュニティほど動きやすくする（重み付き）ことで、ユーザーの up/down vote が「次にどのコミュニティが動くか」へ反映され、唯一の関与手段である vote に意味が増す。
- **cold start 床（+1）**: vote 0 の新規コミュニティも稀に選ばれることを保証し、誰も vote していないコミュニティが永久に沈黙する事態を防ぐ。

## 検討した代替案

- **案A: 全 community 維持 + 生成内容を軽量化**: API コール回数自体は減らせず、コストがコミュニティ数に比例し続けるため不採用。
- **案B: ラウンドロビン（順番に 1 コミュニティ）**: 公平だがエンゲージメントを反映せず、vote というユーザーの唯一の関与が選定に効かない。観察の意味づけが弱いため不採用。
- **案C: 純粋一様ランダムで 1 コミュニティ**: 実装は最小だが、vote を集めたコミュニティが優遇されず「vote に意味がある」体験にならないため不採用。重み付き + 床で B/C の利点（公平な最低保証）を内包する。
- **案D: 重みを `Post.score` / `Comment.score` の累積から算出**: 累積値は時系列の窓を切れず、過去に盛り上がったコミュニティが永久に有利になる。「直近 7 日」の鮮度を反映するため生 `Vote.createdAt` から集計する案を採用。

## 影響（結果）

- 良い影響:
  - API コスト/日がコミュニティ数から独立し、コミュニティを増やしてもコストが頭打ちになる。
  - vote が選定に反映され、ユーザーの関与（up/down vote）に観察上の意味が増す。
- トレードオフ / 注意点:
  - 各コミュニティの更新は間欠的になる（毎定時に全コミュニティが更新されるわけではない）。ホームフィードは引き続き新着順で、定時ごとに更新されるのは選ばれた 1 コミュニティのみ。
  - 重みの式・期間・床値の最適値は運用しながら調整する（`VOTE_WEIGHT_WINDOW_DAYS` 等の定数で調整可能）。
- フォローアップが必要なこと（スコープ外・別 Issue）:
  - 重みの式の動的チューニング（管理画面からの調整）、期間以外の減衰関数、複数コミュニティ同時選定への拡張。
