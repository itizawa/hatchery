# ADR-0005: common パッケージの責務と依存方向

- ステータス: Accepted
- 日付: 2026-05-30
- 関連 Issue: #1

## コンテキスト（背景）

ADR-0001 で `common` を client/server 共通のロジック置き場と定めた。責務と依存ルールを曖昧にすると、UI 依存やサーバ依存が混入して「共通」でなくなり、循環依存や肥大化を招く。境界を明文化する。

## 決定

**`common` は、実行環境に依存しない純粋な TypeScript パッケージとする。**

置くもの:

- **ドメインモデルと型**: 社員（Employee）、シーン（Scene）、メッセージ（Message）、チャンネル、タスク等のドメイン型。
- **ドメインロジック**: 登場メンバー選定、あらすじ要約のルール、ログの整形・直近 N 件の切り出しなど、UI/DB に依存しない純粋関数。
- **Zod スキーマ**: API リクエスト/レスポンスおよびドメインのバリデーションスキーマ。client（型・フォーム検証）と server（入力検証・OpenAPI 生成、ADR-0006）が共有する。
- **共有定数・ユーティリティ**: 環境非依存のもの。

置かないもの:

- React / MUI / DOM などブラウザ依存コード（→ client）
- Express / Prisma / Node 固有 API・DB アクセス（→ server）

依存ルール:

- **依存方向は client → common、server → common の一方向のみ**。`common` は client/server に依存しない。
- `common` は外部ライブラリ依存を最小限にする（Zod など環境非依存のものに限る）。
- ESLint で依存方向（import 制約）を機械的に強制する。

## 理由

- 「共通」を純粋・環境非依存に保つことで、client/server 双方から安全に再利用できる。
- ドメインロジックを `common` に集約すると、UI/DB を立ち上げずに高速なユニットテスト（Vitest）で TDD でき、CLAUDE.md の方針に合う。
- Zod スキーマを `common` に置くことで、検証・型・OpenAPI の単一情報源になり、client/server 間の不整合を防ぐ。

## 検討した代替案

- **型だけを共有（ロジックは各所に重複）**: 実装が二重化し、ほのぼの演出の肝であるドメインルール（登場選定・要約）がずれる。不採用。
- **common に薄い HTTP クライアントまで持たせる**: 環境依存（fetch/Node）が混ざり「純粋」でなくなる。HTTP は client 側（openapi-fetch, ADR-0006）に置く。

## 影響（結果）

- 良い影響: ドメインの単一情報源。高速にテストできる純粋ロジック層が生まれる。
- トレードオフ: 「これは common か client/server か」の判断が都度必要 → 本 ADR の置く/置かない基準で判断する。
- フォローアップ: ESLint の import 境界ルール設定（ADR-0002 の lint 設定に含める）。
