# ADR-0001: monorepo 構成（docs / client / server / common）

- ステータス: Accepted
- 日付: 2026-05-30
- 関連 Issue: #1

## コンテキスト（背景）

Hatchery は SPA フロントエンド（client）と API サーバ（server）からなり、両者でドメインモデル（社員・シーン・メッセージ等）や型を共有したい。さらに設計書／Storybook（docs）も同じリポジトリで管理し、開発ワークフロー（Dark Factory）を 1 リポジトリで完結させたい。

リポジトリを分割（polyrepo）すると、型・ドメインロジックの共有がバージョン公開・取得の手間を伴い、変更の同期がずれやすい。小規模・少人数・AI 主導開発という前提では、横断変更が 1 PR で完結する monorepo が適している。

## 決定

**単一リポジトリの monorepo とし、ワークスペースを 4 つに分ける。**

```
.
├── docs/      # 設計書・ADR・Storybook（ドキュメント基盤）
├── client/    # React SPA（ブラウザ向けフロントエンド）
├── server/    # Express API サーバ（Node.js）
└── common/    # client/server 共通のドメインロジック・型・スキーマ
```

- 各ディレクトリを独立したワークスペース（パッケージ）として管理する（パッケージ名は `@hatchery/client` 等のスコープ付き）。
- 依存方向は **client → common**、**server → common** の一方向のみ。client と server は互いに依存しない。common はアプリ固有パッケージに依存しない（詳細は ADR-0005）。
- `docs` はビルド成果物（Storybook）を生成するワークスペースで、client のコンポーネントと設計 MDX を集約する（詳細は ADR-0007）。

## 理由

- ドメインロジック・型を `common` に集約し、client/server から直接参照できる（公開・取得の手間がない）。
- フロント・バック・共通の横断変更を 1 つの PR・1 つの設計書で扱えるため、Dark Factory ワークフローのゲート（設計レビュー / 実装レビュー）と相性が良い。
- 依存方向を一方向に固定することで循環依存を防ぎ、責務境界を明確化できる。

## 検討した代替案

- **polyrepo（リポジトリ分割）**: サービス境界は明確になるが、型共有のたびに npm 公開やサブモジュール運用が必要で、少人数・AI 主導には過剰。不採用。
- **単一パッケージ（ワークスペース分割なし）**: 最も単純だが、client/server/common の依存方向を機械的に強制できず、責務が混ざりやすい。不採用。
- **3 ワークスペース（docs を独立させない）**: Storybook を client 内に置く案。ドキュメントとアプリのビルド・デプロイ境界が曖昧になるため、`docs` を独立ワークスペースとして分離した。

## 影響（結果）

- 良い影響: 型・ドメインの単一情報源（single source of truth）が `common` に定まる。横断変更が容易。
- トレードオフ: ワークスペース間の依存・ビルド順序の管理が必要 → ADR-0002（pnpm + Turborepo）で対応。
- フォローアップ: ルートに workspace 定義（`pnpm-workspace.yaml`）と共有 tsconfig ベースを置く（ADR-0002）。
