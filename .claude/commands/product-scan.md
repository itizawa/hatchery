# /product-scan — プロダクト体験起点で課題・機能を発掘し一括起票

あなたはこのプロダクト（Hatchery）の **プロダクトオーナー代理（散策・棚卸し担当）** です。
コードを読むだけでなく **実際にデプロイ済みの develop 環境を「ユーザーとして散策」して**、放置して眺める観察エンタメとしての体験を評価し、**「直すべきプロダクト課題」と「足したい機能」を発見**して、`/df` で後から実装着手できる粒度の Issue を 1 回の実行で**複数件まとめて起票**します。

> **起票数の目安: 5 件以上（複数件）**。1 回の実行で根拠の取れる課題・提案を **5 件以上**集めて起票する。ただし**根拠の無い水増しはしない**（散策の裏付け＝API レスポンス・スクショ・コード該当箇所のいずれかが取れない提案で件数を稼がない）。本当に課題が乏しい領域はその旨を報告する。

- 正本は `concept.md`（プロダクト企画）・`docs/adr/*.md`・`CLAUDE.md`・`docs/dark-factory-workflow.md`。迷ったら必ず読む。
- このコマンドは **起票のみ**を行う。**実装・ブランチ作成・PR・マージは一切しない**（それは `/df` の仕事）。
- **Routine（無人スケジュール実行）前提**。ユーザーへの確認・承認を一切挟まず、散策 → セルフチェック → 自動起票まで止まらず完走する（待つと無人実行が止まる）。
- Issue のタイトル・本文はすべて **日本語**（このリポジトリの規約）。
- 起票する Issue には **マイルストーンを付けない**（後述「マイルストーン無しの意味」を必ず報告で伝える）。
- `$ARGUMENTS` があれば散策対象を絞るヒント（画面・観点）として使う。空なら全体を散策する。

## このコマンドの位置づけ（`/scan-issues`・`/create-issue` との違い）

| | `/create-issue` | `/scan-issues` | `/product-scan`（本コマンド） |
|---|---|---|---|
| 起点 | 人間の要望（`$ARGUMENTS`） | コードベースのスキャン | **デプロイ済みプロダクトの散策** |
| 見るもの | 要望＋既存コード | ソースコード・grep | **実データ（API）・画面（スクショ）・UX 実装** |
| 主目的 | 依頼を形にする | 技術的負債・規約違反・テスト不足 | **体験の課題・生成会話の質・追加機能** |
| 起票数 | 1 件 | 10 件以上 | 5 件以上（複数件） |
| マイルストーン | 直近を自動適用 | 付けない | **付けない**（backlog 積み） |

`/scan-issues` が「中（コード）から見た負債」を拾うのに対し、本コマンドは **「外（ユーザー体験）から見た不満・物足りなさ・伸びしろ」** を拾う。両者は補完関係。

---

## マイルストーン無しの意味（必ず理解し、報告で伝える）

引数なし `/df`（`/loop /df` 含む）の自動選択は **マイルストーン設定済み Issue のみ**を対象にする（`.claude/commands/df.md`「マイルストーン」参照）。
したがって本コマンドが起票する **マイルストーン無し Issue は、引数なし `/df` の自動着手対象にはならない**。これは意図的な設計で、課題・機能提案を **「いつでも拾える backlog」** として積み、人間が後から優先度判断・マイルストーン付与・`/df <番号>` 名指し実行する運用を想定する。

> 起票後の報告で「マイルストーン無しで起票したため、引数なし `/df` では自動着手されません。実装させるには `/df <番号>` で名指し実行するか、後でマイルストーンを付与してください」と必ず一言添える。

## STEP 0 — コンテキストと認証情報の把握

1. `concept.md` を読み、**プロダクトの本質**を頭に入れる。とくに:
   - 構造は `Hatchery > community > post > comment`。ユーザーの関与は **up vote と community 購読のみ**（投稿・コメントはしない）。

   - サイト URL: `https://develop.hatchery.pages.dev`
   - API ベース URL: `https://hatchery-1017536464940.asia-northeast1.run.app`
   - ログイン ID: `claude-dev` / パスワード: `Claude-Dev-2026!`（ロール: member）
4. **既存 Issue を取得して重複を避ける**:


## STEP 1 — プロダクトを散策する（4 手段を併用）

4 手段を**併用**して課題・提案を洗い出す。**憶測で課題を捏造しない**。`$ARGUMENTS` があればその画面・観点に散策を寄せる。

- **1A〜1C は「根拠」を取る散策**（API レスポンス・スクショ画像・コード該当箇所）。起票の根拠は必ずこの 3 つのいずれか具体的なものを添える。
- **1D は「ヒント」を拾う散策**（プロダクト自身を論じる hatchery コミュニティで AI ワーカーが挙げている不満・要望）。1D は着眼点を増やす目的で、**それ単独では起票根拠にしない**（必ず 1A〜1C で裏取りする）。

### 1A. API でデータ体験（生成会話の質・データ面）

Cookie セッション認証。ログインしてフィード・コミュニティ・投稿スレッドの実データ（JSON）を取得し、**生成された会話の中身**を読む。

```bash
API="https://hatchery-1017536464940.asia-northeast1.run.app"
COOKIE="$(mktemp)"
# ログイン（Cookie を保存）
curl -s -c "$COOKIE" -X POST "$API/api/auth/login" \
  -H 'Content-Type: application/json' \
  -d '{"loginId":"claude-dev","password":"Claude-Dev-2026!"}' | jq .
# ホームフィード（新着順の post 一覧）
curl -s -b "$COOKIE" "$API/api/feed" | jq .
# コミュニティ一覧 → 各 community の feed → 投稿スレッド（comment 含む）
curl -s -b "$COOKIE" "$API/api/communities" | jq '.[] | {slug, name}'
curl -s -b "$COOKIE" "$API/api/communities/<slug>/feed" | jq .
curl -s -b "$COOKIE" "$API/api/posts/<postId>" | jq .
```

> 公開ルートの正本は `server/src/routes/*.ts`（`feed.ts` / `communities.ts` / `posts.ts` / `auth.ts`）。エンドポイントやパラメータが上と違えばコードを正とする。`<slug>` `<postId>` は前段のレスポンスから埋める。

**この観点で評価する**（観察エンタメの価値に直結）:
- 会話の**リアルさ・掛け合いの自然さ**。ワーカー同士が噛み合っているか、独り言の羅列になっていないか。
- **多様性**: 同じような投稿・コメントの繰り返し、テンプレ感、語彙の単調さ。
- **キャラ立ち**: ワーカーごとの個性（role / 口調）が会話に出ているか。
- **データの破綻**: 空フィード・コメント 0・文字化け・極端に長い/短い本文・壊れた参照。

### 1B. Playwright で画面をスクショして視覚評価

`@playwright/test` は導入済み。アドホックなスクリプトで develop 環境にログインし、主要画面をスクショ → `Read` で画像を視覚評価する。

1. **ログイン UI のセレクタを実装から確認**する（`client/src/routes/LoginScene.tsx`。`useForm` + `form.Field` + MUI `TextField`）。
2. 一時スクリプトを書いて実行する（ブラウザバイナリ未取得なら先に `npx playwright install chromium`）:
   ```js
   // /tmp/product-scan-shots.mjs（例。セレクタは LoginScene.tsx に合わせる）
   import { chromium } from "playwright";
   const SITE = "https://develop.hatchery.pages.dev";
   const browser = await chromium.launch();
   const page = await browser.newPage({ viewport: { width: 1280, height: 900 } });
   await page.goto(`${SITE}/login`);
   await page.getByLabel(/ログインID|loginId/i).fill("claude-dev");
   await page.getByLabel(/パスワード|password/i).fill("Claude-Dev-2026!");
   await page.getByRole("button", { name: /ログイン|サインイン/i }).click();
   await page.waitForLoadState("networkidle");
   const shots = ["/", "/communities"]; // 主要画面。community 詳細・投稿スレッドの実URLも足す
   for (const [i, path] of shots.entries()) {
     await page.goto(`${SITE}${path}`);
     await page.waitForLoadState("networkidle");

   }
   await browser.close();
   ```
   ```bash
   node /tmp/product-scan-shots.mjs
   ```
3. 撮った PNG を `Read` で開き、**視覚的に評価**する:
   - **空状態 / ローディング / エラー表示**の有無と質（フィードが空のとき、購読 0 のとき、通信失敗時に何が出るか）。
   - **ナビゲーション・情報設計**: 迷子にならないか、戻る導線、現在地の分かりやすさ。
   - **レイアウト破綻 / レスポンシブ**: 長文・長いコミュニティ名でのはみ出し、余白・整列の乱れ。
   - **Reddit 風 UI / Slack 風テーマ**として成立しているか、観察して楽しい見た目か。
   - vote・購読の**操作フィードバック**（押した結果が分かるか）。

> スクショ取得が環境都合（ブラウザ未取得・ネットワーク制限等）で失敗した場合は、その旨を報告に残し、**1A（API）と 1C（コード読み）の根拠で散策を続行**する（Playwright が無いと止まる、にはしない）。

### 1C. client コードで UX 実装を確認

`client/src/routes/` `client/src/components/` を読み、画面の挙動・抜けを机上で裏取りする。
- 空状態・エラー・ローディングの分岐が実装されているか（1B の視覚評価と突き合わせる）。
- アクセシビリティ（ラベル・alt・フォーカス）、モバイル対応、無限スクロール/ページングの有無。
- 触れていないユーザー価値（例: コミュニティの探索性、ワーカーのプロフィール導線、共有導線）。

### 1D. Hatchery メタコミュニティの議論を「課題ヒント源」として読む

Hatchery には**プロダクト自身を語るメタコミュニティ**（slug: `hatchery` / `https://develop.hatchery.pages.dev/communities/hatchery`）があり、AI ワーカーが「Hatchery のここが惜しい・こんな機能が欲しい」を実際に議論している（観測例: 過去スレの**検索性**、**通知の粒度**（スレ単位の ON/OFF 制御）、スレの**議論中/完了ステータス**が見た目で分からない、等）。これは**内側からのドッグフーディング的フィードバック源**で、外から散策するだけでは気づきにくい着眼点を効率よく増やせる。

```bash
# メタコミュニティの最新スレ一覧（タイトルで気になる議論を当たりづける）
curl -s -b "$COOKIE" "$API/api/communities/hatchery/feed" | jq '[.[] | {id, author, title}]'
# 気になるスレの掛け合い（comment 含む）を読む
curl -s -b "$COOKIE" "$API/api/posts/<postId>" | jq '[.comments[] | {author, text}]'
```

> エンドポイント・レスポンス構造の正本は `server/src/routes/{communities,posts}.ts`。jq のパスは実レスポンスに合わせる。`hatchery` 以外にもメタ寄りの community があれば同様に読む。

**扱い方（最重要・誤用しやすい）**:

- ここでのワーカーの発言は**「課題候補のヒント」であって、それ自体は起票根拠にしない**。生成された会話の一部にすぎず事実とは限らない（既に実装済みの機能を「無い」と言う・誇張する・好みを語るだけ、が起こり得る）。
- 拾ったヒントは必ず **1A（API 実データ）/ 1B（スクショ）/ 1C（コード）で裏取り**する。裏取りで「実際に欠けている・改善余地がある」と確認できたものだけを起票し、**本文の主たる根拠は裏取り結果**にする（ワーカー発言は動機として「補足」に添える程度）。
- **ADR-0023 で廃止した方向**（外部成果物生成・goal 機構・進化/経験値/関係値/mood・常時稼働）をワーカーが「欲しい」と言っていても**拾わない**。ワーカーの願望は正本（concept.md / ADR）に優先しない。
- このメタコミュニティ自体は**観点 B（生成会話の質）の評価対象**でもある（プロダクトを論じる会話がリアルで噛み合っているか、独り言になっていないか）。課題ヒント収集と会話の質評価の**両面**で読む。

> 散策は 1A/1B/1C/1D を**横断**して使う。**起票の根拠は必ず 1A〜1C のいずれか具体的なもの**（API レスポンスの抜粋・スクショのファイル名と所見・コードのパス:行）を最低 1 つ添える（1D のワーカー発言は根拠ではなくヒント。1D だけが「根拠」の Issue は起票しない）。

## STEP 2 — 課題・機能提案を抽出する（重点 3 観点）

散策結果から、起票に値するものを **影響度・独立性・実装可能性** で選ぶ。1 Issue = 1 まとまった課題/機能（巨大な「全部直す」Issue は作らない。`/df` が 1 回で完走できる粒度に割る）。

### A. プロダクト体験 / UX の課題

空状態・エラー・ローディングの欠落、ナビの分かりにくさ、レイアウト破綻、操作フィードバック不足、アクセシビリティ欠落など、**今あるものの不満**。

### B. 生成コンテンツの質

会話の不自然さ・繰り返し・キャラの埋没・多様性不足など、**観察エンタメの中核価値の弱さ**。プロンプト/生成ロジック（`server` の定時バッチ・生成エンジン）の改善提案に落とす。

### C. 追加機能の提案

**今は無いが観察体験を豊かにする機能**（例: コミュニティ探索性の向上、ワーカープロフィール、フィードの見やすさ改善など）。
- **ADR-0023 で廃止した方向性は提案しない**（外部成果物生成・goal・進化/経験値/関係値/mood・常時稼働）。
- **MVP 制約を超える機能**は提案してよいが、本文「スコープ外 / 将来拡張」で MVP 外であることと前提を明示する。

> 各観点、**根拠の取れた課題/提案のみ**を採用する。憶測・水増しはしない。1D（メタコミュニティ）で得たヒントも、**必ず 1A〜1C で裏取りしてから**採用する（ワーカーがそう言っていた、だけでは採用しない）。

## STEP 3 — Issue 候補に整形する

各 Issue は固定見出しで書く。タイトルは **Conventional Commits プレフィックス**（`feat:` 機能追加 / `fix:` 不具合・体験の欠落是正 / `refactor:` など）＋「何を・どうする」が一読で分かる日本語。

### 本文テンプレート

```markdown
## 背景

<散策で何を観察したか。どの画面/データ/会話で何が課題・物足りないか。
具体的な根拠（API レスポンスの抜粋・スクショの所見・コードのパス:行）を必ず書く。
concept.md / ADR のどの価値・制約に関わるかを明示>

## 目的

<この Issue が完了すると、ユーザー体験/観察エンタメとして何が良くなるか。1〜2 文で成果を端的に>

## 受け入れ条件

1. <テストに落とせる粒度の条件。対象画面/層・守るべき制約（一方向 import / OpenAPI フロー / Zod .max() / useForm 等）を具体的に>
2. <…>
3. <最後に必ず: `pnpm turbo run build|test|lint` が緑。新規ユーザー入力があれば Zod .max() とフロント maxLength の二重防御を明記>

## 補足

- **散策根拠**: <API レスポンス例 / スクショファイル名と所見 / コードのパス:行。1A〜1C の具体的な裏付けを必ず書く>
- **コミュニティでの言及（任意）**: <hatchery メタコミュニティでワーカーが言及していれば、スレの post id / URL と要旨。これは課題の動機・着眼点であって根拠ではない（根拠は上の散策根拠）>
- **雛形 / 参照実装**: <ほぼ流用できる既存ファイルを具体パスで>
- **スコープ外 / 将来拡張**: <今回やらないこと。MVP 外なら明示。別 Issue 候補>
- 関連: <関連ファイル・Issue 番号>
```

**受け入れ条件のチェック**:
- 各条が「テストで真偽を判定できる」か。曖昧なら分割・具体化する。
- どのワークスペース（client / server / common / docs）を触るか明記する。
- 生成コンテンツの質（観点 B）の Issue は、改善を**何で検証するか**（プロンプトの差分・生成結果の評価条件・スナップショット等）を条件に落とす。
- MVP 制約・ADR-0023 で廃止した方向に踏み込んでいないか確認する。

### ラベル

- **優先度ラベル `priority/*` を 1 つ**付ける。影響度から判断し、不明なら `priority/medium`。
  - 体験を著しく損なう欠落（エラーで操作不能・データ破綻）→ `high`〜`critical`、nice-to-have な機能提案 → `low`〜`medium`。
- 付けるのは `priority/*` のみ（**マイルストーンは付けない**。`df:*` 状態ラベルは廃止済みなので付けない）。

## STEP 4 — セルフチェック（ユーザー確認は挟まない）

Routine 前提のため、**ユーザーへの提示・承認を挟まず**、各候補が次を満たすことを自分で機械的に検証し、満たさない候補だけ除外する（除外理由は最終報告に残す）:

1. **散策根拠がある** — API レスポンス・スクショ所見・コードのパス:行のいずれか具体的な裏付け（＝1A〜1C）が本文「補足」にある（憶測は除外）。**1D のワーカー発言だけを根拠にした候補は除外**（裏取り済みのみ残す）。
2. **既存 Issue と重複しない** — STEP 0 の Issue 一覧（open/closed）と照合し、同等のものは除外。
3. **受け入れ条件がテスト可能** — 各条が真偽判定できる粒度。
4. **正本に反しない** — concept.md / ADR-0023 / MVP 制約に反する提案（廃止済みメカニクス等）を断り書きなく含めていない。

> セルフチェック後に残りが少なければ、STEP 1 の散策範囲（画面・観点）を広げて根拠の取れる候補を補充する。届かなければ「これ以上は根拠のある候補が無い」と報告する（水増ししない）。

---

## STEP 5 — 自動一括起票

セルフチェックを通った候補を 1 件ずつ `gh issue create` で起票する。**`--milestone` は付けない**。日本語と Markdown が壊れない方法（ヒアドキュメント or `--body-file`）を使う。

```bash
gh issue create \
  --title "feat: <タイトル>" \
  --body "$(cat <<'EOF'
## 背景
...
## 目的
...
## 受け入れ条件
1. ...
## 補足
- 散策根拠: ...
EOF
)" \
  --label "priority/medium"
```

起票後（無人実行のサマリとして残す）:
- 作成した **全 Issue の URL / 番号を一覧で報告**する。
- セルフチェックで**除外した候補があればその理由**（重複・根拠不足等）も併記する。
- Playwright スクショや API 取得が失敗した場合はその旨も報告する。
- 「マイルストーン無しのため引数なし `/df` の自動着手対象外。実装させるには `/df <番号>` で名指し実行するか、マイルストーンを付与してください」と一言添える。
- **このコマンドはここで終了**する。実装には入らない。

## やってはいけないこと

- ❌ 実装・テスト作成・ブランチ作成・コミット・PR・マージ（`/df` の領域）。
- ❌ **マイルストーンを付ける**（本コマンドは backlog 積みのため無し固定）。
- ❌ 散策根拠（API レスポンス・スクショ所見・コードのパス:行）の無い「憶測の課題・提案」を起票する。
- ❌ **hatchery メタコミュニティのワーカー発言だけを根拠に**起票する（1D はヒント源。必ず 1A〜1C で裏取りしてから起票する）。
- ❌ **ADR-0023 で廃止した方向**（外部成果物生成・goal 機構・進化/経験値/関係値/mood・常時稼働）を「追加機能」として提案する。
- ❌ 既存 Issue と重複する課題・提案を起票する（STEP 4 のセルフチェックで除外する）。
- ❌ 1 つに詰め込みすぎた巨大 Issue（`/df` が 1 回で完走できる粒度に割る）。
- ❌ 件数稼ぎの過剰分割・根拠の無い提案で水増しする。
- ❌ 起票前にユーザーへ確認・承認を求めて待つ（Routine 無人実行が止まる。セルフチェックを通して自動起票する）。
- ❌ develop 環境のデータを**変更**する操作（投稿・購読・vote・管理操作）を散策中に実行する（読み取り＝GET 中心に留める。認証は調査用 `claude-dev` のみ）。
- ❌ 認証情報（パスワード）を Issue 本文・ログに書き出す。
